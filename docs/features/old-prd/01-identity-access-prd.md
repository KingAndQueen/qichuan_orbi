# **产品需求文档: 统一身份与通行证 (Identity & Access)**

状态: Frozen (V7.0) | 优先级: P0 | 最后更新: 2025-11-24

品牌名称: 新智流 (Orbitaskflow)

版本说明: V7.0 MVP Admin \- 定义“组织与设置”管理后台；移除第三方集成；确立“以人为中心”的账号创建与绑定流程。

## **1\. 核心目标 (Core Objective)**

构建一套支持 B2B 多租户管理 和 知识资产继承 的账户体系。

核心是基于业务线（子账号）的资源分配与权限控制，确保企业能精准控制 AI 成本，并实现岗位经验的无缝传承。同时，通过隐私分级机制保护员工的心理安全感。

## **2\. 账户体系架构 (Account Architecture)**

### **2.1 三层模型**

1. **主账号 (Master Account / Tenant)**:  
   * **定位**: 租户管理员，付费主体。  
   * **能力**: 拥有上帝视角，创建和管理子账号，购买/分配订阅席位。  
2. **子账号 (Sub-Account / Seat)**:  
   * **定位**: **业务角色 / 数字工位** (如 "法务-初级审核", "电商-文案组")。  
   * **属性**: 拥有特定的数字员工（工作流）**使用权限**。  
   * **数据**: 存储所有的对话历史记录（是资产的实际载体）。  
3. **员工账号 (User Account)**:  
   * **定位**: 操作员 (具体自然人，如张三)。  
   * **属性**: **独立的登录凭证 (账号 \+ 密码)**。  
   * **行为**: 被**绑定**到某个子账号上进行操作。一个员工可属于多个子账号（切换身份）。

### **2.2 权限与可见性规则 (Visibility Rules)**

#### **A. 资源分配逻辑 (Resource Allocation)**

1. **租户级席位池 (Tenant Pool)**: 主账号购买的数字员工席位（License）归属于**企业租户**，形成一个全局资源池。  
2. **业务级授权 (Sub-Account Access)**: 主账号只需勾选\*\*“哪些子账号可以使用此工作流”\*\*（开关式授权），无需手动切分席位数量。  
3. **视图隔离**: 只有被授权的子账号下的员工，才能看到对应的工作流入口。
4. **数字员工职级（Agent Access Levels）**：
* **背景**: 区分“查看者”与“执行者”，控制风险。  
* **L1 \- 观察员 (Viewer)**: 仅有**只读**权限（如读取知识库、搜索），回答问题，**严禁**执行写操作（如发邮件、改数据库）。  
* **L2 \- 起草者 (Drafter)**: 可以生成草稿（Draft Email, Draft Code），但必须由**人类点击确认**后才能发送或生效（Human-in-the-loop）。  
* **L3 \- 执行者 (Editor)**: \[高风险/需特批\] 允许在特定白名单范围内自动执行写操作。

#### **B. 对话可见性状态 (Conversation Visibility)**

1. **🔒 私有 (Private)**:  
   * **默认状态**。  
   * **可见范围**: 仅当前操作员工 \+ 主账号管理员（审计用）。  
   * **同组同事**: **不可见**。  
2. **🌐 团队公开 (Team Public)**:  
   * **主动操作**。员工认为此对话有价值，手动切换开关。  
   * **可见范围**: 同子账号下的所有员工（只读）。  
3. **📂 离职归档 (Archived Asset)**:  
   * **触发条件**: 员工与子账号解绑（离职/转岗）。  
   * **系统行为**: 该员工产生的所有“私有”对话自动转为“只读归档”。  
   * **可见范围**: 继任者及同子账号员工（只读）。

## **3\. 交互细节规范 (UX Specifications)**

### **3.1 登录与初始化**

* **登录页**: 极简设计，仅展示 **新智流 (Orbitaskflow) 品牌 LOGO**、Slogan 和 **账号/密码输入框**。  
* **密码策略**: 低门槛（\>=8位即可）。  
* **忘记密码**: 文案引导：“如忘记密码，请联系您的企业管理员重置。”  
* **登录流转**: 登录成功后，系统自动识别用户的所属身份。  
  * **单身份用户**: 直接进入智能工作台，无需额外操作。  
  * **多身份用户**: 展示“选择业务身份”页面，用户点击卡片（如“法务组”或“项目组”）后进入相应的工作环境。

### **3.2 隐私合规提示 (Privacy Compliance)**

* **资产归属告知**: 系统必须在用户首次登录或《用户协议》摘要中，明确告知用户：“**工作期间产生的 AI 对话属于企业资产，离职后将留存给团队。**”

## **4\. 管理后台需求 (Admin Portal) \[NEW\]**

目标: 提供原生、闭环的组织架构管理能力。  
入口: 侧边栏底部 \[身份卡片\] \-\> \[组织设置\] (原三方管理)。

### **4.1 成员管理 (Members) \- 以人为中心**

* **F4.1.1 成员列表**:  
  * **展示**: 姓名、登录账号(Email/手机)、当前状态(激活/禁用)、**所属部门(子账号)**。  
  * **操作**: 编辑、重置密码、禁用。  
* **F4.1.2 添加成员 (Add Member)**:  
  * **交互**: 点击 \[+ 添加成员\] 弹出侧滑抽屉。  
  * **基础信息**: 输入姓名、登录账号、初始密码。  
  * **岗位绑定 (Binding)**:  
    * **必填项**: 提供 \[所属部门/子账号\] 下拉单选/多选框。  
    * **逻辑**: 管理员在创建人的同时，直接将其分配到“法务部”或“销售部”。若尚未创建部门，提供 \[+ 新建部门\] 快捷入口。  
* **F4.1.3 账号全生命周期**:  
  * **入职**: 创建并告知初始密码。  
  * **转岗**: 修改“所属部门”字段，系统自动迁移权限。  
  * **离职**: 点击 \[禁用\]，立即阻断登录，触发 F2.2.B 的“离职归档”逻辑。

### **4.2 部门与子账号管理 (Departments/Sub-accounts)**

* **F4.2.1 部门列表**:  
  * **定义**: 这里管理的每一个 Item 就是一个 **“子账号 (Sub-account)”**。  
  * **展示**: 部门名称、成员数、已分配 Agent 数。  
* **F4.2.2 部门设置**:  
  * **创建**: \[+ 新建部门\] (如 "上海分公司-法务")。  
  * **编辑**: 修改名称。  
  * **删除**: 仅当部门下无成员且无资产时允许删除。

## **6. 安全与访问控制规则 (Security & Access Rules)**

*本章节定义系统的登录安全策略与账号生命周期规则。*

* **S1. 访问控制策略 (Access Control Strategy)**:
  * **会话有效期 (Session Lifecycle)**:
    * **活跃保持**: 采用 **7 天滑动过期** (Rolling Expiration) 机制。只要用户在 7 天内有活跃操作，无需重新登录。
    * **强制重登**: 出于合规要求，无论活跃与否，**每 30 天** 必须强制重新验证身份一次。
  * **防暴力破解 (Brute-force Protection)**:
    * **规则**: 单个账号在 **10 分钟内连续输错 5 次** 密码。
    * **惩罚**: 账号暂时锁定 **15 分钟**，并在登录页显示明确的倒计时提示。

* **S2. 批量快速入职 (Bulk Onboarding)**:
  * **场景**: 针对小微企业 (Small Business) 管理员需一次性添加 10-50 名员工的场景，替代繁琐的单次添加。
  * **交互方案**: 在“添加成员”抽屉中增加 **[批量添加]** 选项卡。
    * **方式 A (极简模式)**: 提供一个大文本框，允许管理员直接 **复制粘贴** 类似 "姓名, 手机号/邮箱" 的多行文本（支持 Excel 直接复制粘贴），系统自动解析并预览。
    * **方式 B (标准模式)**: 提供 **[下载模板]** 和 **[上传文件]** 按钮，支持标准的 CSV/Excel 导入。
  * **初始凭证**: 批量导入的账号默认分配统一的**临时密码**（由管理员在界面设置），员工首次登录后**强制要求修改密码**。
* **S3. 安全审计与登出 (Audit & Logout)**:
  * **用户行为审计**: (保持原样)
  * **[NEW] 系统调用审计 (Syscall Audit)**: 
    * 记录 Agent 进程发起的所有高危操作。
    * **字段**: `Process ID`, `Agent Name`, `Syscall Type` (e.g., `fs.write`, `net.connect`), `Target` (e.g., `/salary.csv`, `google.com`), `Result` (Allow/Deny)。
