## **1\. 一句话摘要**
作为[TERM-WS-001]工作台即操作系统，用户通过[TERM-WS-004]超级输入框以自然语言与文件下达指令，发起[TERM-IA-005]工作流并由[TERM-IA-007]智能员工在[TERM-WS-002]进程中基于[TERM-WS-003]虚拟文件系统执行长时任务，以[TERM-WS-005]服务端驱动界面呈现可视化进度与自适应交付结果，同时支持 IDE 级布局管理与多业务线导航，形成可追踪、可协作的端到端工作体验。

## 2. 目标与指标（Goals & Metrics）
### 2.1 产品目标（Goals）
- G0-1（基线目标）：用户启动[TERM-WS-002]进程后，即使离开页面/网络中断/关闭浏览器，返回后仍能继续操作推进该任务（不要求用户重来），且进程状态与历史产出保持一致，不会“丢任务、卡死、需要重来”。
- G0-2（基线目标）：同一任务的结果呈现稳定一致：系统可按任务返回内容自适应展示（组件/表格/图表/表单/[TERM-WS-008]智能追问），且历史任务在不同客户端版本下均可正常展示，不出现“无法渲染/展示错乱/同任务不同呈现”的问题。
- G1：让用户清楚“任务现在在哪里、还要多久、下一步要做什么”：以可理解的状态与时间信息呈现[TERM-WS-002]进程进度，并在需要人工介入时明确提示并进入“[TERM-WS-010]需要协助”。
- G2-1：让“[TERM-WS-012]交付物可直接用”：每次关键输出至少包含 1 个可直接使用的[TERM-WS-012]交付物形态（如文档/表格/表单/组件），用户可直接复制/导出或进入[TERM-WS-006]智能编辑器精修，而不是只得到一段文本。
- G2-2：让“下一步可一键推进”：每次关键输出同时提供清晰的[TERM-WS-011]交付动作（如继续生成、补充信息、选择范围、确认参数、批准执行），用户无需反复用自然语言“猜怎么继续”。
- G3：让“下达指令”更简单：以[TERM-WS-004]超级输入框完成自然语言 + 文件/附件的一站式输入，并能明确选择或识别[TERM-IA-005]工作流后启动执行。
- G4：让“交付可精修可复用”：提供[TERM-WS-006]智能编辑器，用于对[TERM-WS-012]交付物进行编辑/版本对比/导出与复用，减少在聊天消息中反复复制粘贴。
- G5：让“找得到、切得动”：工作区提供统一的导航入口，用户能快速定位并切换历史[TERM-WS-013]工作会话、正在执行/已完成的[TERM-WS-002]进程，以及对应的结果与导出记录。
- G6：让“出问题也好用”：当[TERM-WS-002]进程失败、卡住或进入[TERM-WS-010]需要协助时，用户能看到清晰原因与可执行的下一步（重试/继续/提交信息/联系支持），并能一键提交问题反馈；同时支持人员可基于最小诊断信息快速定位问题，减少用户等待与重复操作。
### 2.2 成功标准（体验结果）
- 成功标准 S1：对用户而言，结果展示“跟任务走”：系统会根据任务返回内容自动选择合适的呈现方式（组件/图表/表单/[TERM-WS-008]智能追问等），无需用户理解内部实现细节，也不会因客户端升级/降级而出现“同任务不同呈现、或无法渲染”。（规则：R4、R5）
- 成功标准 S2：用户能完成一次完整闭环：输入（文字+文件）→ 发起并执行[TERM-IA-005]工作流（由[TERM-IA-007]智能员工执行）→ 看到可理解的进度与可操作的阶段性结果 → 进入[TERM-WS-006]智能编辑器完成精修 → 导出 / [TERM-IA-011] 团队公开，并可追溯引用来源。（规则：R1、R6、R7、R8、R9、R10、R15）
- 成功标准 S3：长时任务体验可控：用户离开或断网后返回，仍能看到任务继续推进；当任务需要人工补充信息或确认时，会明确进入“[TERM-WS-010]需要协助”，用户补齐信息后可继续执行直至完成。（规则：R1、R2、R11）
- 成功标准 S4：每次关键输出至少提供 1 种“可直接使用”的交付方式（复制格式化/导出/打开[TERM-WS-006]智能编辑器继续），且用户不需要手动整理才能使用。（规则：R8、R10）
- 成功标准 S5：每次关键输出至少提供 1 个明确的[TERM-WS-011]交付动作（按钮/表单/确认项等），用户点击后可继续推进任务。（规则：R9、R13）
- 成功标准 S6：当任务失败/中断/卡住时，界面会给出可理解的原因提示与至少 1 个可执行动作（重试/继续/提交信息/联系支持），且用户可在当前界面一键提交问题反馈。（规则：R12、R11）
### 2.3 指标定义与统计口径（Metrics，统计口径不强制事件名）
#### 北极星指标（North Star Metric）

- 指标名：可交付完成率
  - 对应成功标准：S2、S4
  - 一句话定义：发起的[TERM-WS-002]进程中，已完成且发生至少一次[TERM-WS-011]交付动作的比例。
  - 存在理由：衡量 Workspace 是否把任务带到“可交付/可使用”的闭环阶段，避免仅用“[TERM-WS-002]进程完成”高估用户价值（对应目标：G2-1）。
  - 指标类型：ratio
  - 分析对象：process（[TERM-WS-002]进程）
  - 时间粒度：day
  - 时间窗：统计周期（如自然日/自然周/自然月；按报表选择）
  - 过滤条件：Workspace 模块内有效用户操作；排除内部测试账号、自动化回放、异常数据。
  - 维度（可选）：客户端版本 / [TERM-IA-005]工作流 / [TERM-IA-002]子账号（TBD）
  - 数据来源：进程状态与[TERM-WS-011]交付动作相关行为事件/表（TBD）
   - 计算口径：
      | 组成项 | 定义 | 去重口径 | 过滤条件 |
      |---|---|---|---|
      | 分子 | 已完成且发生至少一次[TERM-WS-011]交付动作的[TERM-WS-002]进程数 | 按[TERM-WS-002]进程唯一标识去重 | 同上 |
      | 分母 | 发起的[TERM-WS-002]进程数 | 按[TERM-WS-002]进程唯一标识去重 | 同上 |
      - 公式：分子 / 分母

## 5. 用户旅程（Key Flows）
### 5.1 旅程 1：发起任务（文件 + 指令 + 工作流）
- 触发：用户在 [TERM-WS-004] 超级输入框输入指令并携带附件/粘贴内容。
- 用户目标：快速选择正确 [TERM-IA-005] 工作流并发送任务；可中断/停止并保留草稿。
- 主流程：
  1. 用户拖拽/点击/粘贴添加附件（显示独立附件栏卡片）。（规则：R6）
  2. 选择 [TERM-IA-005] 工作流（下拉列表，仅展示已订阅工作流）。（规则：R7）
  3. 知识/定制包确认：系统解析该工作流的步骤级 [TERM-WS-018] 信息需求，汇总本次将访问的 [TERM-WS-017] 企业知识源；若存在 [TERM-WS-020] 智能体定制包引用，同步展示其名称与版本；用户可确认/拒绝/更换可用来源；若拒绝导致必需信息需求无法满足，则进程进入 [TERM-WS-010] 需要协助。（规则：R16、R18）
  4. Shift+Enter 换行编辑长文本；输入内容以纯文本方式编辑并原样保留。（规则：R6）
  5. 点击发送后按钮变为“停止（■）”，用户可随时停止/取消并恢复草稿。（规则：R1）
- 分支/约束：
  - 当前 [TERM-WS-013] 工作会话进行中切换工作流需自动开启新 [TERM-WS-013] 工作会话，避免上下文污染。（规则：R7）
- 失败与恢复：
  - 文件失败：附件卡片变红 + Tooltip，用户可移除或重试。（规则：R12）
  - 网络中断：断线重连后可继续操作推进 [TERM-WS-002] 进程，并保持状态一致（异常体验规范见 WS-012）。（规则：R1、R12）
- 关联需求：WS-001、WS-002、WS-003、WS-004
### 5.2 旅程 2：任务执行与结果交付（生成式界面 + 编辑器）
- 触发：[TERM-IA-007] 智能员工开始流式输出文本/组件，或判定产出为 [TERM-WS-012] 交付物。
- 用户目标：实时看到进度与结构化结果；需要时进入 [TERM-WS-006] 智能编辑器精修并导出。
- 主流程：
  1. 自适应流式反馈：优先在 1s 内提供首个可理解反馈（如“执行中/正在确认/正在检索”等状态或步骤反馈）；若为直接文本生成，则以文本流打字机特效呈现且文本首字延迟 < 1s。（规则：R2）
  2. 若服务端下发 [TERM-WS-005] 服务端驱动界面：以结构化交互卡片呈现需要用户确认/补信息/审批的内容，并将用户操作结果回传至对应 [TERM-WS-002] 进程以推进下一步。（规则：R4、R11）
  3. 若[TERM-IA-007] 智能员工判定为 [TERM-WS-012] 交付物：展示“在编辑器中打开”入口；用户点击后右侧展开 [TERM-WS-006] 智能编辑器。（规则：R8、R10）
  4. 结果证据化：当结果依赖企业知识源时，输出必须提供可引用来源（至少到文档/页面/条目级），并允许用户点击查看来源摘要；若无可用证据，必须明确提示“不足以给出结论”，并进入 [TERM-WS-010] 需要协助收集缺失信息。
  5. 用户编辑、触发行内意图（局部润色/缩短/翻译/自定义 Prompt），系统做局部更新。（规则：R10）
  6. 需要导出时，从编辑器右上角导出 PDF/Word/Markdown 或复制格式化内容。（规则：R10）
- 分支/约束：
  - 当 [TERM-IA-007] 智能员工输出了不可识别/不支持的组件类型时，客户端必须安全降级并保证 [TERM-WS-012] 交付物可用。（规则：R4、R5）
- 失败与恢复：
  - 生成失败：消息底部显示 Regenerate，且已产生的阶段性内容与[TERM-WS-012] 交付物仍可访问。（规则：R12）
  - 权限不足/信息缺失：[TERM-WS-002] 进程进入 [TERM-WS-010] 需要协助状态，并提供可执行选项。（规则：R11）
- 关联需求：WS-005、WS-006、WS-007、WS-008、WS-009、WS-010、WS-011
### 5.3 旅程 3：[TERM-WS-013] 工作会话管理、隐私与共享
- 触发：用户创建新对话、切换 [TERM-WS-009] 临时对话、执行 [TERM-IA-011] 团队公开、管理[TERM-WS-013] 工作会话列表。
- 用户目标：控制隐私、快速定位历史、后台任务可控。
- 主流程：
  1. 新建对话进入零态并展示动态快捷指令；懒创建 [TERM-WS-013] 工作会话（首次发送才入库）。（规则：R1）
  2. [TERM-WS-013] 工作会话列表按时间分组，Hover 显示更多菜单（重命名/删除/置顶可选）。（规则：R14）
  3. 默认 [TERM-IA-010] 私有；点击分享按钮确认后变为 [TERM-IA-011] 团队公开（只读）。（规则：R15）
  4. 开启 [TERM-WS-009] 临时对话后顶部/输入框出现明显提示；内容不入库且禁用团队公开。（规则：R15）
- 分支/约束：
  - 历史可回看与可定位：已完成/已失败的 [TERM-WS-002] 进程必须可在对应 [TERM-WS-013] 工作会话中再次定位与打开。（规则：R3、R14）
- 失败与恢复：
  - [TERM-WS-002] 进程失败/卡住：必须提供可行动反馈（重试/继续/补充信息/提交反馈），且既有[TERM-WS-012] 交付物仍可复用。（规则：R12）
- 关联需求：WS-012、WS-013、WS-001

- 一句话描述：工作台支持用户通过 [TERM-WS-004] 超级输入框以自然语言发起任务，将执行过程组织为可追踪的 [TERM-WS-002] 进程，并以 [TERM-WS-003] 虚拟文件系统承载上下文与产出。（规则：R1、R2、R3）

  - AC-WS-001-1：Given 一个 [TERM-WS-002] 进程 Running，When 断线重连，Then 可恢复继续操作推进该 [TERM-WS-002] 进程，并保持进程状态与历史产出一致。

  - AC-WS-001-3：Given 同一 [TERM-WS-013] 工作会话内存在一个进行中的 [TERM-WS-002] 进程，When 用户刷新页面/重新打开该 [TERM-WS-013] 工作会话，Then 系统应自动定位到该进行中的 [TERM-WS-002] 进程并允许继续操作推进（无需重新发起新 [TERM-WS-002] 进程）。

#### WS-004：混合输入能力与生成控制
- 一句话描述：[TERM-WS-004] 超级输入框支持混合输入并提供生成控制（Stop/恢复草稿），确保用户可控地发起与中止执行。（规则：R6）

- 权限/边界要点：

#### WS-005：自适应流式反馈（Streaming Feedback）
- 一句话描述：对 [TERM-WS-002] 进程的响应提供“首个可理解反馈”（状态/步骤/文本流三选一）并在长任务中持续更新，保证等待可感知、可控。（规则：R2）
- 关联旅程：5.2
- 主要交互：
  - 首个可理解反馈（First Meaningful Feedback）：用户发送后，优先在 1s 内在消息区呈现首个可理解反馈（例如“已受理/执行中/正在确认/正在检索”或第一条步骤反馈）；不要求在所有任务中都在 1s 内产出最终答案文本。（规则：R2）
  - 文本流呈现：当任务进入“直接生成文本”路径时，以文本流方式逐步呈现（打字机特效可选），并尽量缩短文本首字延迟。（规则：R2）
  - 长任务持续反馈：当任务需要较长时间（例如需要外部检索/工具调用/等待用户确认）且短时间内无法持续产出文本时，消息区需持续呈现可理解的执行状态或步骤更新，避免“长时间无变化”的空等待。（规则：R2）
- 权限/边界要点：
  - [TERM-WS-002] 进程状态与反馈必须可理解，且不得伪造确定性时间承诺或虚构进度。（规则：R2）
- 验收标准：
  - AC-WS-005-1（直接文本生成）：Given 正常网络，When 发起“直接生成文本”任务，Then 文本首字延迟 < 1s（在可观测链路上验证）。
  - AC-WS-005-2（非直接文本生成/长任务）：Given 正常网络，When 发起需要外部检索/工具调用/用户确认的任务，Then 在 1s 内必须出现首个可理解反馈（状态或步骤之一），且在长任务期间不得出现用户无感知的长时间空白等待。

#### WS-009：版本与差异（Version & Diff）
- 一句话描述：对 [TERM-WS-006] 智能编辑器中的内容与关键输出提供快照版本与差异对比，并通过默认保留策略控制版本数量，保证可追溯与可回滚。（规则：R10、R5）
- 关联旅程：5.2
- 主要交互：
  - 版本类型：
    - 微版本（Micro Revisions）：用于高频编辑过程中的短期撤销/恢复，不等同于长期可回滚快照。
    - 快照版本（Snapshots）：用于修订对比与回滚的“可追溯版本”。
  - 快照生成（MVP）：在“有意义事件”发生时生成快照版本：
    - 用户导出（PDF/Word/Markdown）或复制格式化内容
    - [TERM-IA-007] 智能员工一次写入/补丁应用完成（apply patch/write 完成）
    - 用户结束一段编辑（停止输入并稳定 5 分钟）后自动生成 1 个快照
  - 默认保留策略（MVP）：对每个 [TERM-WS-012] 交付物的快照版本，默认保留“最近 30 天”或“最近 100 个快照”（取更小者）；超出部分自动清理。
  - 关键版本保留：用户可将某个快照标记为“保留/命名版本”，被标记的版本不参与自动清理。
  - 更新反馈：当 [TERM-IA-007] 智能员工更新编辑器内容后，顶部短暂显示 ✅ 已更新 Toast。
  - 差异高亮：修订模式下，绿色背景标记新增，红色删除线标记移除。
  - 上下文管理：[TERM-IA-007] 智能员工生成回答时仅读取编辑器当前最新版本（Current State）。
- 权限/边界要点：
  - 对同一 [TERM-WS-002] 进程阶段性结果，多端/多版本呈现需稳定一致；降级不得影响 [TERM-WS-012] 交付物可用。（规则：R5）
- 验收标准：
  - AC-WS-009-1（快照生成）：Given 发生导出/复制格式化内容、一次 [TERM-IA-007] 智能员工写入完成，或用户停止输入并稳定 5 分钟，When 事件完成/条件满足，Then 自动生成 1 个快照版本且可在历史中查看。
  - AC-WS-009-2（保留策略）：Given 对同一 [TERM-WS-012] 交付物生成超过 100 个快照或跨度超过 30 天，When 查看历史，Then 未标记的快照按“30 天或 100 个（取更小者）”自动清理，且标记为“保留/命名版本”的快照仍存在。
  - AC-WS-009-3（差异对比）：Given 选择任意两个快照进入修订模式，When 查看差异，Then 新增/删除按规则高亮展示；同时当内容被 [TERM-IA-007] 智能员工更新后出现 ✅ 已更新 Toast。
#### WS-010：反馈、引用与 [TERM-WS-008] 智能追问（Feedback / Citations / Follow-up）
- 一句话描述：支持消息级反馈、引用溯源与 [TERM-WS-008] 智能追问，促进结果可追溯与下一步可行动。（规则：R9、R13）

- 主要交互：
  - 评价与反馈（Per-Message）：消息卡底部 Action Bar 与复制并列

    - 👎 点踩：图标变红；可选弹出 [不准确] [不安全] [格式错误]
  - 引用与溯源（Citations）：

    - Click：滚动到附件区高亮文件或打开来源网页
  - [TERM-WS-008] 智能追问（[TERM-IA-007] 智能员工驱动）：

    - 形态：消息卡片下方独立胶囊按钮行

- 权限/边界要点：
  - 每次关键输出必须提供至少 1 个 [TERM-WS-011] 交付动作；智能追问仅作为降低继续成本的快捷入口，不得成为唯一继续路径。（规则：R9、R13）
- 验收标准：
  - AC-WS-010-1：Given 服务端未提供任何[TERM-WS-008]智能追问项，When 本轮消息输出结束，Then 不展示智能追问入口与列表。
  - AC-WS-010-2：Given 有 citation，When Hover/Click，Then 显示摘要并定位到来源。
#### WS-011：[TERM-WS-007] 推理步骤面板（Thought Transparency）
- 一句话描述：以 Reasoning Steps 折叠面板展示推理过程并在失败时暴露卡点以便用户介入。（规则：R2、R12）
- 关联旅程：5.2

#### WS-013：隐私与共享（Privacy）
- 一句话描述：工作会话默认 [TERM-IA-010] 私有，用户可显式确认开启 [TERM-IA-011] 团队公开；[TERM-WS-009] 临时对话不入库且禁用团队公开。（规则：R15）
- 关联旅程：5.3
- 主要交互：
  - 隐私默认值：所有新[TERM-WS-013] 工作会话默认 🔒 [TERM-IA-010] 私有，仅当前用户（及租户管理员审计）可见
  - 团队公开：
    - 入口：顶部 Header 右上角 [🌐 分享] 按钮
    - 操作：点击弹出确认框“是否将此 [TERM-WS-013] 工作会话设置为 [TERM-IA-011] 团队公开？”
    - 变更：确认后图标变为“🌐 [TERM-IA-011] 团队公开”，同子账号/业务组同事可只读查看
  - [TERM-WS-009] 临时对话模式（Temporary/Incognito）：
    - 触发：侧边栏临时对话开关
    - 视觉反馈：Header 变深灰/特殊色，显示“临时模式 - 内容不保存”；输入框水印“当前对话历史将在刷新后消失”
    - 数据逻辑：消息仅在内存流转，不写入 PostgreSQL 历史记录表
    - 无历史：刷新或新建对话后彻底销毁不可恢复
    - 功能限制：禁用团队公开
- 权限/边界要点：
  - 团队公开必须由用户显式确认后才生效；临时对话不得入库且必须禁用团队公开并提示“内容不保存”。（规则：R15）
- 验收标准：
  - AC-WS-013-1：Given 临时模式开启，When 刷新或新建对话，Then 历史不可恢复且数据库无记录。
  - AC-WS-013-2：Given 临时模式开启，When 点击分享，Then 分享入口应被禁用。
#### WS-014：企业知识源（[TERM-WS-017] Enterprise Knowledge Source）
- 一句话描述：当工作流需要企业既有知识时，系统必须让用户清晰可控地选择/授权本次执行将访问的企业知识源，并在运行时可追溯。
- 关联旅程：5.1 / 5.2
- 主要交互：
  - 访问声明：在启动前或首次触发前展示“本次将访问的企业知识源列表”（名称 + 范围摘要 + 权限状态）。
  - 拒绝与替代：用户可拒绝某来源；若导致必需信息缺失则进入 [TERM-WS-010] 需要协助，提示“请授权/上传/手工输入/终止”。
  - 可追溯：在进程详情中可回看本次使用过的知识源与访问结果概览（不强制暴露实现细节）。
- 权限/边界要点：
  - 必须遵循 IA 的可见性与授权边界：无权限来源不得被静默访问。
- 验收标准：
  - AC-WS-014-1：Given 工作流存在必需的 [TERM-WS-018] 信息需求，When 用户启动执行，Then 系统必须展示将访问的 [TERM-WS-017] 企业知识源，并允许用户拒绝或更换。
  - AC-WS-014-2：Given 用户拒绝导致必需信息需求无法满足，When 继续执行，Then 进程必须进入 [TERM-WS-010] 需要协助并提供替代路径（授权/上传/手工输入/终止）。

#### WS-015：工作流信息需求与场景化检索（[TERM-WS-018] + [TERM-WS-019]）
- 一句话描述：工作流/步骤必须可声明信息需求，并强绑定场景化检索配置档案以提升稳定性；不得依赖通用检索让智能员工“自由发挥”。
- 关联旅程：5.1 / 5.2
- 主要交互：
  - 信息需求声明：工作流包必须能描述每一步需要的企业信息（必需/可选、新鲜度/时间窗、证据要求、允许来源）。
  - 检索配置绑定：每类信息需求必须指向一个 [TERM-WS-019] 检索配置档案；缺失时必须显式降级为人工补充/上传，而非静默输出。
  - 证据输出：当信息需求要求证据时，交付结果必须返回可引用来源。
- 验收标准：
  - AC-WS-015-1：Given 一个步骤声明了 [TERM-WS-018] 信息需求并绑定 [TERM-WS-019]，When 触发检索，Then 系统必须按该档案执行检索/过滤/重排，并在结果中输出可引用来源。
  - AC-WS-015-2：Given 绑定的检索配置档案缺失/执行失败，When 继续执行，Then 系统不得静默给结论，必须进入“需要协助”收集替代输入或提示终止。

#### WS-016：智能体定制包（[TERM-WS-020] Agent Policy Pack）
- 一句话描述：工作流可引用企业的智能体定制包以固化口径与写法；定制包必须版本化、可展示、可回滚。
- 关联旅程：5.1 / 5.2
- 主要交互：
  - 展示：在启动前或详情面板展示定制包名称与版本；同一进程内不得无提示切换版本。
  - 升级：升级仅允许具备权限的角色触发；升级必须有变更说明与回滚入口。
- 验收标准：
  - AC-WS-016-1：Given 工作流引用 [TERM-WS-020]，When 用户启动执行，Then 必须展示定制包名称与版本。
  - AC-WS-016-2：Given 定制包升级发生，When 再次执行同工作流，Then 必须明确提示版本变化并允许回滚到上一个可用版本（若存在）。

- 风险 1：[TERM-WS-005] 服务端驱动界面 的 Schema 迭代导致前后端兼容风险。

### 10.3 WS-015：[TERM-WS-005] 服务端驱动界面 的版本化与安全渲染（Now）
- 描述（What）：
  - [TERM-WS-005] 服务端驱动界面 在下发“可渲染的结构化结果”时，必须携带以下信息类型（不限定字段命名）：
    1) **版本标识**：用于前后端兼容判断。
    2) **组件类型**：前端据此选择对应的展示组件。
    3) **展示数据**：组件所需的内容与配置（例如表格数据、图表数据、表单字段等）。
    4) **可交互动作列表**：用户可点击/提交的动作，以及动作的必要输入。
    5) **安全策略标识**：用于判断该组件与动作是否允许在当前企业/岗位身份/权限下执行。
  - 组件白名单（Approved Components）：
    - 前端只允许渲染“已审核通过”的组件集合；未在集合内的组件一律不渲染。
    - 示例（保持原意，仅作示例）：仪表盘卡片、审批表单、表格、图表、文件差异视图。
  - 兼容策略（保证迭代可控）：
    - 当 UI 描述包含“前端不认识的新增信息”时：前端应安全忽略，不影响已支持部分的渲染。
    - 当 UI 描述要求渲染“前端不支持的组件类型/版本”时：前端必须降级为安全的文本呈现（例如 Markdown/纯文本）并提示“该组件暂不支持”，而不是尝试渲染不受控内容。
  - 安全原则（动作必须可控、可追溯）：
    - 任意交互动作回传给服务端时，必须携带：
      - **动作标识**（用于指向用户点击的是哪一个动作）；
      - **输入摘要/签名信息**（用于防篡改与审计追溯，不要求在 PRD 中规定 hash 算法）。
    - 所有动作在执行前必须经过策略判断：允许则执行，拒绝则阻断并给出可理解的提示；同时记录留痕，满足企业审计需求。
- 最小验收：
  - AC-WS-015-1：当服务端下发未知的组件类型时，前端必须降级为安全提示，不可渲染任意 HTML 或执行任意脚本。
  - AC-WS-015-2：当用户触发任意交互动作时，回传必须包含动作标识与输入摘要/签名信息，并且该动作的允许/拒绝结果可被审计追溯。
