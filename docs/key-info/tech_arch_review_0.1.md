# TECH-ARCH Review 结论汇总（逐步达成一致后写入）

> 规则：本画布只记录“已达成一致”的结论；讨论中的假设/分歧点只在对话框里讨论，不写入这里。

---

## 0. 评审范围与目标（已对齐后写入）

- 评审对象：TECH-ARCH-FULLSTACK-001（Fullstack Architecture v3.9）
- 评审目标：验证当前架构是否能支撑 Orbitaskflow 的长期业务与产品演进，并给出需要冻结的不变量、必须拍板的关键 trade-off、以及必须验证的实验清单。

---

## 1. 冻结为系统不变量（Keep）

- **K-001｜隔离与授权采用“防御纵深（Dev-time + Run-time）”**
  - **Dev-time（开发期）**：通过 CI 规则与自动化测试强制检查隔离上下文的传递与使用，尽可能在合并前消灭“漏传/漏写隔离条件”的缺陷。
  - **Run-time（运行期）**：对关键资源与关键动作提供系统级兜底（DB 强制隔离 / 统一 PEP 执行点 / 必要的能力令牌），确保即使出现实现缺陷也难以越权或跨主账号访问。
  - 目标：将“账号上下文串联/跨主账号访问/越权”从开发规范问题升级为系统级不变量。

- **K-002｜Zero Trust 的架构分层（PDP/PEP）作为长期基座**
  - 在网关与服务边界设置 PEP（执行点）统一拦截与校验；策略决策集中在 PDP（决策点）以支持审计、演进与一致性。

- **K-003｜Capability Token 仅用于高风险/高成本/跨边界动作（最小集合）**
  - 普通 CRUD 不强制 capability；仅对“系统调用级”动作启用，以控制复杂度并保证可撤销/可审计。

## 2. 需要拍板的关键决策（Decide / ADR Backlog）

- **ADR-001｜业务与交付路线冻结：China-led validation + Regionized delivery（国内合规交付 + 海外商业验证并行）**
  - 冻结结论：
    - 场景与商业价值论证以中国市场为主（市场营销 / 财富规划 / 软件开发三条业务线）。
    - 每个场景一旦达到可复制的产品化成熟度，即并行推进海外市场的付费意愿与价值认可验证。
    - 交付形态采用“区域化部署 + 数据驻留默认就地”的架构：中国大陆用户采用国内合规可控的本地云/本地交付；海外（非中国大陆）默认云为 **Google Cloud Platform（GCP）**，其它云厂商可作为后续扩展选项。
  - 关键约束：
    - 跨区域/跨法域的数据流必须显式建模为能力（可开关、可审计、可追踪），不得默认为可跨境。
    - 云厂商、模型供应商、身份源、支付等依赖必须保持可替换（Provider Abstraction），避免将某一区域的特定选择写死到核心域模型中。

- **ADR-002｜交互与界面形态冻结：WebSocket First + Generative UI（主路径）**
  - 冻结结论：
    - **WebSocket First（执行期主通道）**：工作台/执行期的交互与状态流转以 WebSocket 作为主通道。
    - **HTTP First（管理面）**：管理面与配置面保持 HTTP 为主。
    - **SSE/轮询作为降级与补偿**：用于单向通知、弱实时场景，或 WebSocket 不可用时的 fallback。
    - **Generative UI 作为核心能力但必须契约化**：服务端下发 UI 意图/结构需具备 schema 版本、客户端能力协商与回退策略，避免后端变更导致客户端不可用。
    - **事件封装优先标准化**：执行期事件流采用统一的 envelope（可参考 CloudEvents 思路），确保可审计、可演进与跨端一致性。

- **ADR-003｜控制面/执行面分离 + 异步任务引擎冻结（Control Plane / Execution Plane）**
  - 冻结结论：
    - **控制面（Go / Platform Core）**：承载系统不变量与治理能力（身份与隔离上下文、PDP/PEP、订阅/额度、审计、核心对象元数据与状态机、对外 API/WS 连接治理）。
    - **执行面（Python / Agent Runtime Hub）**：承载可变的智能执行与生态接入能力，统一覆盖：
      - **LLM Providers**（多模型供应商调用与路由）；
      - **MCP**（作为 Client/Host 接入外部工具与上下文提供者）；
      - **A2A / Remote Agents**（接入外部 Agent 并进行协作调用）；
      - **API Connectors**（将外部系统/服务以标准适配器方式纳入编排）。
    - **异步任务引擎（Queue/Worker）**：长任务与重计算在执行面异步化；任务语义以“可重试/可幂等”为默认前提，并由控制面状态机与审计链路对齐。
  - 关键约束：
    - 治理逻辑不得散落在各类 Connector/Adapter 中：所有跨边界动作必须经由控制面给出的策略/票据/能力约束后再执行。
    - 外部 Agent/工具默认不可信：执行面调用必须携带隔离上下文并受 PEP 与 capability 约束，确保可审计、可追踪、可撤销。

- **ADR-004｜异步任务与事件处理语义冻结：At-least-once + Idempotency（默认）**
  - 冻结结论：
    - **默认语义**：所有异步任务与事件处理按 **At-least-once** 设计，必须接受“可能重复投递/重复执行”的现实。
    - **默认要求**：所有可重试动作必须具备 **幂等性（Idempotency）**，重复执行的最终效果等价于执行一次。
    - **副作用保护**：涉及外部副作用（例如扣费/额度扣减、写第三方系统、通知发送、授权分发等）必须具备幂等保护或等价机制，避免重复执行导致重复副作用。
  - 备注：更强语义（例如 exactly-once）可作为后续可选增强，但不作为系统默认依赖。

- **ADR-005｜统一事件 Envelope 作为 SSOT 冻结（版本化 + 兼容策略）**
  - 冻结结论：
    - 系统内所有事件流（WebSocket 事件、异步任务事件、审计/计量事件、以及对外 MCP/A2A/API 触发的事件）必须使用**统一 Envelope**。
    - Envelope 必须**版本化（schema version）**并具备明确的**兼容策略**（默认禁止 breaking changes，至少保持 backward compatible）。
    - 事件契约必须有可版本化的“单一事实来源（SSOT）”（例如以 AsyncAPI/等价方式管理 message/channel/schema），并纳入代码仓库与发布流程。
    - Envelope 设计优先保持 **CloudEvents-compatible**（可映射 CloudEvents 核心字段），以便复用生态工具与提升跨系统互操作性。

- **ADR-006｜全链路可观测性与上下文传播冻结（W3C Trace Context + 统一 Propagation）**
  - 冻结结论：
    - 采用 **W3C Trace Context** 作为跨服务/跨语言的默认 trace 传播格式（`traceparent` / `tracestate`）。
    - 所有边界必须提取/注入 trace context：HTTP、WebSocket（建连与每条事件）、异步消息（queue/job/envelope）、以及对外 MCP/A2A/API 调用。
    - 日志与链路必须可关联：关键日志事件应携带可关联的 trace 标识（例如 `trace_id` / `span_id` 或等价字段）。
    - 业务上下文字段的传播必须受控：可使用标准化 baggage 思路，但禁止携带敏感/个人信息，并限制键集合与大小。

- **ADR-007｜数据隔离拓扑冻结：Shared DB + DB-enforced Isolation（默认）**
  - 冻结结论：
    - 默认采用 **Shared DB / Shared Schema** 的数据拓扑，通过 `master_account_id`（以及必要时的 `scope/sub_account_id`）对所有可持久化对象进行分区。
    - 对关键资产与关键写入路径采用 **DB-enforced Isolation** 作为运行期兜底（例如 Postgres RLS），确保即使应用层遗漏隔离条件也难以发生跨主账号访问。
    - 允许后续对极少数强监管/大客户/特殊性能场景演进到混合模型（Bridge/Isolated DB）作为**例外路径**，但不改变默认主路径。
  - 关键约束：
    - 任何绕过数据访问层/策略执行链路的脚本、批处理、运维工具同样必须注入主账号隔离上下文，并受审计与可观测性约束。

- **ADR-008｜主账号隔离上下文契约冻结（Context Contract SSOT）**
  - 冻结结论：
    - 系统内所有边界（HTTP、WebSocket 建连与事件、异步消息/任务、DB 会话变量注入、Redis/VFS 命名空间、以及对外 MCP/A2A/API 调用）必须遵循同一份**主账号隔离上下文契约（Context Contract）**。
    - Context Contract 作为系统级 **SSOT** 管理，与 ADR-005 的事件契约同等级别，纳入代码仓库与版本控制。
    - Context Contract 的“最小必需集合”（概念级）至少包含：
      - **身份与隔离域**：主账号 / 子账号（可选）/ 主体（用户/员工/外部 Agent）；
      - **追踪与关联**：trace/correlation（与 ADR-006 对齐）；
      - **权限与约束引用**：必要时携带 capability/授权票据引用（与 K-003 对齐）。
    - 每个边界必须具备自动化**提取/注入/校验**能力：缺失或不一致必须在关键路径上 fail-closed，并产生可审计、可观测的信号。
  - 关键约束：
    - 全程使用项目 SSOT 的“主账号/子账号/员工账号”等产品语言，不引入“租户”表述。

- **ADR-009｜外部 Agent/工具接入安全模型冻结：Zero Trust 默认（默认不可信）**
  - 冻结结论：
    - 任何外部接入（MCP / A2A / API Connectors）默认视为**外部主体**，不获得隐式权限与环境信任。
    - 外部接入的入口与出口必须经由 **PEP** 执行点统一拦截与校验，并由 **PDP** 做策略决策（允许/拒绝/降权/路由），确保口径一致可演进。
    - 对外调用的高风险/跨边界动作必须使用短期、最小权限、**可撤销**的授权载体（capability/票据），并具备紧急切断能力。
    - 所有外部调用必须绑定主账号隔离上下文与 trace context（与 ADR-006/008 对齐），产生日志与审计信号，确保可追溯与可归因。

- **ADR-010｜外部接入能力描述冻结：机器可读 + 版本化 + 可验证（SSOT）**
  - 冻结结论：
    - 所有外部接入对象（MCP / A2A / API Connectors / Remote Agents）必须提供**机器可读的能力描述**（Capability Descriptor / Agent Card / Tool Manifest）。
    - 能力描述必须包含（原则级）：能力列表、调用协议与端点、输入/输出契约（JSON Schema/OpenAPI/等价）、版本号与兼容策略、安全需求（认证方式、最小权限范围、速率/配额、是否需要 capability）。
    - 能力描述作为系统级 **SSOT** 管理，纳入代码仓库与版本控制，并可被 PDP/PEP 用于策略决策、路由/兼容与自动化校验（CI + 运行时）。
    - 对第三方未提供完整描述的情况，允许以“适配层补齐描述”的方式接入，但不得绕过上述 SSOT 与可验证要求。

- **ADR-011｜可信发布者校验冻结：外部接入必须可验证来源与防篡改（Signature/Attestation）**
  - 冻结结论：
    - 任何外部接入对象（Connector/Remote Agent/MCP Server）及其能力描述（Descriptor/Agent Card/Manifest）必须具备可验证的**签名或来源证明**，并与平台维护的**可信发布者**策略/列表匹配。
    - 校验至少覆盖关键生命周期点：**安装 / 启用 / 升级**；校验失败必须 **fail-closed**（禁止启用/升级）。
    - 校验结果必须产生可审计、可观测的信号（与 ADR-006/008 对齐），支持追溯“由谁发布、何时启用、是否被撤销”。
    - 对第三方未提供签名/证明的情况，允许通过“适配层代签/补齐证明”进行过渡，但不得绕过可信发布者校验与审计要求。

- **ADR-012｜按主账号维度的默认限流/配额冻结（Rate Limit / Quota，PEP 强制执行）**
  - 冻结结论：
    - 平台必须支持并默认启用：**按主账号维度**的限流与配额，至少覆盖关键入口：HTTP、WebSocket 消息、异步任务入队、对外 MCP/A2A/API 调用。
    - 限流/配额必须由 **PEP** 执行点强制执行，并可与订阅/授权额度体系联动（不同套餐/额度对应不同上限）。
    - 默认限流策略需允许短时突发（burst），以平滑体验并防止短时抖动造成误伤。
    - 超限必须产生可观测/可审计信号（与 ADR-006/008 对齐），并支持追溯“在何主账号上下文、何入口、何资源维度发生限额”。

- **ADR-013｜计量事件（Metering）冻结：高成本动作必须事件化、可审计、可重放**
  - 冻结结论：
    - 平台必须将**计量事件**视为一等公民：对高成本/高风险/跨边界动作必须产生日志化、可持久化的计量事件。
    - 计量事件必须绑定主账号隔离上下文，并可与 trace context 关联（与 ADR-006/008 对齐），支持“按主账号归集、对账、追溯与归因”。
    - 计量事件必须纳入统一事件体系（与 ADR-005 的 envelope/版本化/兼容策略一致），避免出现不可解释的“黑盒扣减”。
    - 计量事件应支持重放校验与聚合再计算，以支撑订阅/额度扣减一致性与争议处理。

- **ADR-014｜授权额度变更冻结：统一额度账本 + 状态机（唯一扣减路径、可补偿、可审计）**
  - 冻结结论：
    - 所有授权额度的变更（发放、扣减、冻结、释放、过期、退款/回滚）必须通过统一的**额度状态机**执行，禁止多条口径并存，形成**唯一扣减路径**。
    - 每次额度变更必须写入可持久化、可审计的**额度账本记录**（可追溯），并能与主账号隔离上下文、trace、计量事件关联（与 ADR-006/008/013 对齐）。
    - 必须具备明确的**补偿/冲正**语义以应对重试与外部失败，避免人工“改数”作为常态手段。
    - 任何“额度状态变更 + 事件发布/通知”必须遵循一致性护栏（例如 outbox/inbox 等价模式），避免扣减与事件不一致。

- **ADR-015｜额度并发语义冻结：Reservation/Hold → Commit/Release（默认）**
  - 冻结结论：
    - 额度状态机必须支持最小并发控制语义：**预留（Reservation/Hold）→ 确认（Commit）/释放（Release）**，以避免并发双扣与透支。
    - 对“长耗时/跨边界/可能失败重试”的动作，默认采用 **Reserve → Commit/Release**；对“短链路、低风险、可瞬时完成”的动作允许白名单直扣。
    - 预留/确认/释放必须落账本并可审计，可与计量事件与 trace 关联（与 ADR-006/013/014 对齐）。
    - 预留必须具备明确的生命周期治理（例如 TTL/超时自动释放），避免长期占用额度。

- **ADR-016｜额度预留凭证冻结：Reservation Token 作为执行前置条件（PEP 强制校验）**
  - 冻结结论：
    - 对任何需要消耗授权额度的动作，必须先由控制面创建 **Reservation/Hold** 并返回 **reservation_token（预留凭证）**。
    - 执行面（Python）与对外 MCP/A2A/API 调用必须携带 reservation_token；**PEP** 必须校验 token 的有效性、未过期/未撤销、绑定的主账号隔离上下文一致性（与 ADR-008 对齐）、以及额度类型/上限/资源维度匹配。
    - 缺失 reservation_token 或校验失败：关键路径必须 **fail-closed**（禁止继续执行），并产生可审计、可观测信号（与 ADR-006/008 对齐）。
    - reservation_token 必须可关联到计量事件与额度账本/状态机记录（与 ADR-013/014/015 对齐），并可与 trace 关联（与 ADR-006 对齐）。

- **ADR-017｜异步任务额度闭环冻结：任务携带 reservation_token，Worker 执行前强校验，结束必须 Commit/Release**
  - 冻结结论：
    - 任何需要消耗授权额度的异步任务载荷必须携带 **reservation_token**，并同时携带主账号隔离上下文与 trace context（与 ADR-006/008/016 对齐）。
    - Worker 在执行前必须进行等价于 PEP 的校验：token 有效、未过期/未撤销、绑定主账号一致、资源维度匹配；校验失败必须 **fail-closed** 并产生审计/可观测信号。
    - 任务执行结束必须闭环：成功必须 **Commit**；失败/取消/超时必须 **Release**（或进入明确的补偿/冲正流程），禁止预留长期悬挂占用。
    - Commit/Release 与事件发布/通知必须遵循一致性护栏（outbox/inbox 等价模式，与 ADR-014 对齐），避免扣减与事件不一致。

- **ADR-018｜异步幂等与去重冻结：统一 Idempotency/Dedupe Key（按主账号域，入队与执行两侧可去重）**
  - 冻结结论：
    - 所有可能被重试/重复投递的异步任务必须具备统一的 **idempotency_key / dedupe_key**，并按**主账号域**生效（同一主账号内去重，不同主账号互不影响）。
    - 去重/幂等保护必须至少在**执行侧（Worker/Handler）**强制生效；入队侧（Enqueue）应尽可能做前置去重以降低重复任务进入系统。
    - 对存在外部副作用的动作（第三方写入、通知发送、授权分发、扣减/发放等），必须绑定同一 idempotency_key，确保重复执行不会产生重复副作用。
    - idempotency_key 必须可关联主账号隔离上下文、reservation_token、计量事件与 trace（与 ADR-006/008/013/016 对齐），并产生可审计信号以解释“去重/重放”的结果。

- **ADR-019｜一致性护栏冻结：Transactional Outbox（或等价模式）+ Idempotent Consumer 默认**
  - 冻结结论：
    - 所有需要可靠事件发布的写路径（尤其：额度账本/状态机、授权分发、订阅实例、审计/计量事件、对外通知等）必须使用 **Transactional Outbox**（或等价模式），确保“业务状态变更”与“事件记录”在同一事务内落地。
    - Outbox 事件必须使用统一 Envelope（与 ADR-005 对齐），并携带主账号隔离上下文、trace，以及与 idempotency_key/dedupe_key 的关联字段（与 ADR-006/008/018 对齐）。
    - 事件消费端必须默认具备 **Idempotent Consumer** 能力（例如 inbox/dedupe 表或等价机制），至少能基于 event_id 或 dedupe_key 确保“重复投递不产生重复副作用”（与 ADR-004/018 对齐）。
    - 禁止绕过 outbox 的双写路径进入主干；如需例外必须显式白名单并具备等价的一致性保证。

- **ADR-020｜关键对象版本语义冻结：object_version（单调递增）+ OCC，版本写入事件 Envelope**
  - 冻结结论：
    - 对关键对象的状态变更必须提供 **object_version**（单调递增）与 **乐观并发控制（OCC）** 语义，避免并发写入导致丢失更新。
    - 更新关键对象时必须携带“期望版本（expected_version）”或等价条件；不匹配时必须拒绝或触发上层重试/冲突处理流程。
    - 每次关键对象状态变更产生的事件必须在统一 Envelope 内携带 **object_id + object_version**（以及必要的状态字段），用于消费端乱序检测与缺口识别。
    - 消费端在处理事件时必须能基于 object_version 检测乱序/缺失，并触发补拉/等待/降级等策略（策略细节可迭代，但检测能力必须存在）。

- **ADR-021｜关键读路径新鲜度语义冻结：Read-your-writes + Monotonic Reads（可验证）**
  - 冻结结论：
    - 对关键对象的关键读路径，平台必须提供**可验证的新鲜度语义**（至少覆盖同一主账号/同一主体的会话内）：
      - **Read-your-writes**：写后读必须能读到自己写入的结果；
      - **Monotonic Reads**：连续读取同一对象时读到的版本不得倒退。
    - 缓存/客户端必须使用 **object_version（或等价 ETag）** 进行再验证与回源策略（例如条件请求），以低成本校验“是否变更”。
    - 当检测到版本倒退/乱序（读到更小的 object_version）时，必须触发回源/等待/降级等纠正路径（策略细节可迭代，但检测与纠正入口必须存在）。

- **ADR-022｜WebSocket 推送一致性冻结：携带 object_version + 单连接单调 + 缺口检测与 Pull-to-repair**
  - 冻结结论：
    - 所有 WebSocket 下发的关键对象状态变更事件必须携带 **object_id + object_version**（与 ADR-020 对齐）；建议同时携带连接级/会话级 **stream_seq** 以快速检测缺口。
    - 对同一 object_id，在同一连接内服务端推送的 object_version 必须保持**单调不倒退**（与 ADR-021 对齐）。
    - 任一方检测到缺口/乱序（例如收到更高版本但本地未连续）必须触发 **Pull-to-repair**：客户端发起补拉（按对象/范围/last_seen_version），回源拉齐后再继续消费推送。
    - 断线重连必须携带 last_seen 的 stream_seq/版本信息；若无法恢复则必须走补拉修复路径，确保状态最终可校验可修复。

- **ADR-023｜WebSocket 断线恢复冻结：有限回放窗口（Replay Window）+ 超窗强制补拉**
  - 冻结结论：
    - 服务端必须提供**有限窗口**的事件回放能力（replay window），以 **stream_seq** 作为恢复锚点：客户端重连携带 last_seen_stream_seq，窗口内回放缺失事件。
    - 超出回放窗口时服务端必须明确返回“不可恢复”，客户端必须触发 **Pull-to-repair**（按对象/范围/last_seen_version）进行补拉修复（与 ADR-022 对齐）。
    - 回放窗口大小必须**可配置**（按环境/套餐/成本策略），并对回放/超窗/补拉触发产生可观测与可审计信号（与 ADR-006/013 对齐）。

- **ADR-024｜回放缓冲存储形态冻结：独立轻量事件缓冲 + 仅存 Envelope 最小集（禁止业务表扫描回放）**
  - 冻结结论：
    - Replay Window 的回放缓冲必须**独立于业务表**，采用轻量事件缓冲（Redis Streams / 专用事件表 / 专用流服务等均可）；**严禁**通过扫描业务表/更新时间推导来实现事件回放。
    - 回放缓冲只允许存储统一 Envelope 的**最小字段集合**（含主账号隔离上下文、trace、stream_seq、object_id/object_version 等必要字段），避免大 payload 导致成本与延迟失控。
    - 回放缓冲必须具备明确的**窗口化治理**（按时间/大小裁剪与保留策略），并与 ADR-023 的“超窗不可恢复 → 强制补拉”语义一致。
    - 回放命中率、超窗率、补拉触发率必须可观测与可审计（与 ADR-006/013 对齐），用于持续权衡体验与成本。

- **ADR-025｜回放窗口 Schema 兼容冻结：Envelope 与 ADR-005 同源版本化；窗口内禁 Breaking Change，破坏则降级补拉**
  - 冻结结论：
    - 回放缓冲中的事件必须使用与 ADR-005 相同的 **Envelope schema_version** 与兼容策略（同一份 SSOT 管理）。
    - 回放窗口覆盖的时间窗内禁止 **breaking change**；窗口内的变更必须至少保持 backward compatible（更严格策略可迭代，但不得破坏回放期内兼容）。
    - 一旦发生 breaking change（或无法保证兼容），服务端必须将该连接的“回放”降级为“不可恢复”，强制客户端走 **Pull-to-repair** 补拉最新状态（与 ADR-022/023 对齐）。
    - 回放缓冲仅存 Envelope 最小集（与 ADR-024 对齐），Schema 演进必须遵循渐进式扩展/收缩路径，避免回放窗口内出现不可解释的不兼容。

- **ADR-026｜补拉修复治理冻结：Pull-to-repair 必须按主账号限流/配额/计量 + 分页/范围约束**
  - 冻结结论：
    - Pull-to-repair（补拉修复）接口必须纳入 ADR-012/013：按**主账号维度**限流与配额（PEP 强制执行），并产生日志化、可审计的计量事件以归因与对账。
    - 补拉修复必须强制**分页**与**范围约束**（按对象/版本区间/时间窗/cursor 等），并设置最大 page size，禁止“无限范围拉取”。
    - 对异常高频补拉（持续缺口/恶意）必须具备降级/熔断路径（例如延迟重试、强制快照、临时拒绝等），并产生可观测信号（与 ADR-006 对齐）。
    - 所有补拉修复请求必须携带主账号隔离上下文与 trace context（与 ADR-006/008 对齐）。

- **ADR-027｜补拉响应可验证元数据冻结：object_version/ETag + data_source + last_event_stream_seq（若有）**
  - 冻结结论：
    - Pull-to-repair（补拉修复）的响应必须返回最小可验证元数据：**object_version（或等价 ETag）**、**data_source**（如 origin/cache/replica 等枚举/等价字段），以及在系统维护 stream_seq 时返回 **last_event_stream_seq**（作为恢复锚点，与 ADR-022/023 对齐）。
    - 客户端必须记录“已见版本/已见序号”，用于 WebSocket 缺口检测、断线恢复锚点与避免读到倒退版本（与 ADR-021/022/023 对齐）。
    - 若服务端无法提供部分元数据（例如无 stream_seq），必须明确降级口径（至少返回 object_version/ETag，并提供等价的后续修复路径）。

- **ADR-028｜客户端协议版本闸门冻结：Web 优先的最小版本协商（MVP 不做复杂能力矩阵）**
  - 冻结结论：
    - MVP 阶段客户端以**网页端**为主，但必须考虑“多版本并存”（浏览器缓存/CDN/长连接/多标签页）带来的兼容风险，因此需要**最小版本协商闸门**。
    - WebSocket 建连必须完成**协议版本协商/声明**（例如子协议版本或首条 hello 消息携带 client_protocol_version），服务端维护可接受的最小/最大版本范围。
    - 版本不匹配必须**显式拒绝或降级**：优先提示客户端刷新/升级；必要时降级到 HTTP/SSE 的可用路径（与 ADR-002 的回退策略对齐），避免“连接成功但运行期随机崩溃”。
    - MVP 阶段**不冻结复杂能力矩阵**（feature-by-feature negotiation）；仅冻结“协议版本闸门”。未来多端/多能力扩展时，再在该闸门之上演进为能力协商。

- **ADR-029｜协议演进规则冻结：小版本向后兼容；Breaking Change 必须大版本并由版本闸门阻断**
  - 冻结结论：
    - 协议类 SSOT（至少包含：事件 Envelope、主账号隔离上下文契约、WS 推送/回放/补拉契约）必须采用**清晰的版本语义**进行演进管理。
    - 默认只允许**向后兼容**的小版本演进（minor/patch）：旧网页端在可接受窗口内必须可用。
    - 任何 **breaking change** 必须提升为**大版本（major）**，并通过 ADR-028 的协议版本闸门**显式阻断**旧客户端（提示刷新/升级或走降级路径），禁止隐式破坏导致运行期随机崩溃。
    - 推荐采用渐进式扩展/收缩（expand/contract）路径承接不兼容需求：先兼容扩展，迁移完成后再收缩移除。

- **ADR-030｜版本闸门拒绝信息冻结：机器可读 reason_code + min_supported_version + recommended_action**
  - 冻结结论：
    - 当服务端因协议版本闸门拒绝 WebSocket 建连/运行时中止时，必须返回**机器可读**的拒绝信息（可在握手阶段拒绝或建连后关闭），至少包含：
      - **reason_code**（如 PROTOCOL_VERSION_UNSUPPORTED/TOO_OLD 等枚举）；
      - **min_supported_version**（服务端要求的最小可用版本）；
      - **recommended_action**（如 refresh/upgrade/fallback_http_sse 等枚举）。
    - 网页端必须能基于上述信息稳定展示“刷新/升级/降级”提示，避免白屏与不可解释错误（与 ADR-028/029 的版本演进规则对齐）。

- **ADR-031｜Web 强制更新策略冻结：recommended_action=refresh 必须触发统一 cache-bust/强制更新路径**
  - 冻结结论：
    - 当服务端返回 recommended_action=refresh 时，网页端必须执行**统一的强制更新策略**，确保用户刷新后一定能获取到新版本资源，避免“刷新仍命中旧包”的死循环。
    - 入口 HTML（或等价启动资源）必须采用**可再验证**的缓存策略（禁止长缓存），以便及时获取最新版本。
    - 静态资源必须采用**内容指纹化（content-hash）**与长缓存策略（可 immutable），以实现“版本切换即缓存失效”。
    - 强制更新必须具备明确的 cache-bust 触发方式（例如带 build_id 的入口参数/版本化入口路径等），具体实现可迭代，但必须保证刷新链路可用且可验证。
    - 若未来引入 Service Worker，必须补充 SW 更新/接管策略以保证强制更新的确定性（MVP 可不启用 SW，但需预留扩展位）。

- **ADR-032｜服务端版本可发现性冻结：server_build_id/version endpoint + WS hello 下发（贯穿日志与事件）**
  - 冻结结论：
    - 服务端必须提供**机器可读**的版本可发现能力，用于网页端判断是否需要强制更新、以及用于排障归因：
      - 提供一个稳定的版本查询入口（如 /meta/version 或等价 endpoint），返回至少包含 **server_build_id**（以及可选的 git_sha / build_time / deploy_region 等非敏感元数据）。
      - WebSocket 建连成功后的首条 hello（或等价握手确认消息）必须下发 **server_build_id**，以支持长连接运行期间的版本判断。
    - 网页端在关键交互链路（尤其是 WS 建连与关键错误处理）必须记录 client_build_id 与 server_build_id；当检测到不一致或被 ADR-030 指示 refresh 时，触发 ADR-031 的强制更新策略。
    - server_build_id 必须贯穿到关键可观测与审计信号：日志、关键事件 Envelope（与 ADR-005/006 对齐）应可关联到 server_build_id，以便快速定位“哪次部署/哪一批实例”导致的问题。
    - 版本查询入口必须采用可控缓存策略（避免长缓存导致误判）；字段集合必须受控且不包含敏感信息（与 ADR-006 的传播约束对齐）。

- **ADR-033｜HTTP 关键 API 版本闸门冻结：写接口必须受协议版本治理 + 机器可读拒绝信息**
  - 冻结结论：
    - 除 WebSocket 外，**HTTP 管理面与关键 API**（尤其关键写接口）也必须纳入协议版本治理，避免 WS 之外仍发生不兼容调用。
    - 服务端必须在关键写接口上校验 client_protocol_version（或等价版本标识）；不满足最小版本时必须 **fail-closed**。
    - 拒绝时必须返回与 ADR-030 同口径的**机器可读**信息（reason_code + min_supported_version + recommended_action），以支持网页端稳定提示与降级。
    - MVP 阶段可先覆盖“关键写接口”与“管理面关键路径”；读接口可逐步纳入，扩展到未来多端时保持同一套版本治理语义（与 ADR-028/029/030 对齐）。

- **ADR-034｜对话资产/交付物存储边界冻结：元数据入 Postgres；大对象入对象存储；主账号命名空间隔离；默认静态加密（可选 CMEK/KMS）**
  - 冻结结论：
    - **Postgres 仅存结构化元数据与索引**：关键对象（至少包含 [TERM-IA-009] 对话资产、[TERM-WS-012] 交付物）的归属与状态必须落在 Postgres（受 RLS/主账号隔离约束），并包含 master_account_id、object_version、大小/指纹（sha256 等）、以及 object_uri/object_key 等引用字段（与 ADR-007/008/020/021 对齐）。
    - **大 payload/文件内容进入对象存储**：文本大字段、文件、导出包、版本快照等大对象不得长期内联存于业务表；应存入对象存储，并以元数据引用方式在 Postgres 中关联。
    - **主账号命名空间隔离（强制）**：对象存储 key/prefix 必须包含 master_account_id（必要时包含 scope/sub_account_id）；上传/下载必须走服务端鉴权与签名 URL（或等价受控访问），确保 PEP/审计/计量可落地（与 ADR-008/009/012/013 对齐）。
    - **默认静态加密**：对象存储必须默认启用静态加密；需要更强密钥治理时允许启用 KMS/CMEK（作为可选增强，不强制 MVP 全量流程）。
    - **MVP 允许阈值化内联**：允许对“小内容”在可控阈值内内联存储于 DB；超过阈值必须落对象存储（阈值为可配置策略，原则固定）。

- **ADR-035｜对象存储受控访问冻结：Signed URL（短 TTL + 最小权限）+ 签发审计/计量（按主账号归因）**
  - 冻结结论：
    - 对象存储的上传/下载必须统一走服务端签发的 **Signed URL / Presigned URL**（或等价受控访问），服务端在签发前必须完成主账号隔离上下文校验与权限校验（与 ADR-008/009 对齐）。
    - Signed URL 必须采用**短 TTL**并遵循**最小权限**原则：限定单对象、单动作（read/write）、可选限制内容类型/大小/校验和等；禁止授予可枚举/可跨目录/可跨主账号的泛权限。
    - Signed URL 必须**按需生成**且不得作为长期凭证持久化存储；如需复用必须重新签发并重新审计。
    - 每次签发必须产生日志化、可审计、可对账的事件，并纳入计量体系（与 ADR-013 对齐）：至少包含 master_account_id、object_key、action、ttl、trace、client_build_id/server_build_id 等关键字段，用于按主账号归因与成本治理（与 ADR-006/012/032 对齐）。

- **ADR-036｜删除语义冻结：默认软删 + 异步硬删闭环（DB + 对象存储一致删除，硬删可审计可证明完成）**
  - 冻结结论：
    - **默认软删（Soft Delete）**：删除操作必须首先软删并立即在产品可见性上生效（查询/列表/搜索/导出均不可见）。软删必须保留最小墓碑（tombstone）元数据（如 deleted_at/deleted_by/delete_reason 可选），并遵循关键对象版本语义（object_version 单调递增，与 ADR-020/021 对齐）。
    - **异步硬删（Hard Delete）闭环**：硬删作为后台任务执行，必须同时覆盖 Postgres 元数据与对象存储内容（与 ADR-034/035 对齐），并允许安全重试；硬删处理必须幂等（重复执行不产生误删与异常）。
    - **可审计、可证明完成**：软删/硬删全过程必须产生日志化、可持久化的审计/计量事件（与 ADR-013 对齐），可追溯到 master_account_id、对象标识、对象存储 key、发起主体与 trace。硬删完成必须形成“完成证据”（例如 hard_deleted_at 或 delete_completion_event），用于证明对象存储删除成功/或确认不存在。
    - **扩展位预留但不强制 MVP**：回收站/延迟硬删窗口、legal hold 等治理能力可后续迭代补齐，但软删墓碑结构需允许扩展。

- **ADR-037｜删除保留与SLA冻结：软删保留窗口（N 天可配置）+ 硬删完成 SLA（T 可配置）+ 删除链路可观测指标**
  - 冻结结论：
    - **软删保留窗口必须存在且可配置**：软删对象在进入硬删前必须受窗口化治理（N 天为策略参数，MVP 可给默认值但不得写死在代码逻辑中）。
    - **硬删完成 SLA 必须存在且可配置**：从软删发生到 DB + 对象存储完成硬删并写入完成证据（与 ADR-036 对齐）必须有目标上限（T 为策略参数），并具备告警阈值。
    - **删除链路必须纳入可观测指标与告警**（与 ADR-006/013 对齐），至少覆盖：
      - hard-delete backlog（积压数量/队列长度）；
      - backlog age（最老任务年龄，P95/P99）；
      - 硬删成功率/失败率/重试次数；
      - time-to-hard-delete（从软删到完成证据的端到端耗时分布）；
      - 按主账号聚合的删除量与异常峰值（用于归因与成本/风险治理）。

- **ADR-038｜对象存储生命周期策略冻结：Lifecycle/Retention 与软删窗口一致 + 策略统一下发（禁止漂移/绕过）**
  - 冻结结论：
    - 对象存储的 **Lifecycle/Retention** 策略必须与 ADR-037 的软删保留窗口（N 天）保持一致口径，避免出现“平台仍在保留窗口内但对象已被存储侧提前删除”的不一致与审计断裂。
    - Lifecycle/Retention 必须基于 ADR-034 的主账号命名空间（prefix/master_account_id）进行可控应用（或等价策略），降低误删与跨主账号影响面。
    - 策略配置必须由平台统一的 IaC/配置源下发并版本化管理；禁止绕过平台手工修改导致策略漂移（drift）。如检测到漂移必须产生告警并具备回滚/纠正路径。
    - 必须在运行事实上承认并处理生命周期策略变更的生效延迟/异步性：监控与删除 SLA 需要考虑对象存储侧删除是异步执行的现实（与 ADR-037 对齐）。

- **ADR-039｜导出语义冻结：导出包入对象存储（短 TTL + 可撤销）+ 导出动作计量/审计 + 导出结果主账号强绑定**
  - 冻结结论：
    - 任何导出结果（导出包/下载工件）必须作为“大对象”落入对象存储并通过 Postgres 元数据引用关联（与 ADR-034 对齐），禁止长期内联存储于业务表。
    - 导出包下载必须统一走 ADR-035 的受控访问路径：服务端签发 Signed URL（短 TTL + 最小权限），并支持“可撤销”（例如删除导出对象/更换访问策略/签发密钥轮换等等价手段），避免链接泄露后长期可用。
    - 导出对象的 key/prefix 必须包含 master_account_id（必要时包含 scope/sub_account_id），导出任务/导出元数据必须受 RLS/主账号隔离约束，禁止跨主账号复用导出结果（与 ADR-007/008/034 对齐）。
    - 导出属于高成本/高风险动作：导出生成、Signed URL 签发、以及下载行为（若可观测）必须产生日志化、可审计、可对账的计量事件（与 ADR-013/032/035 对齐），用于按主账号归因与成本治理。
    - 导出格式与范围必须可版本化（export_schema_version 或等价字段）：MVP 可先提供最小可用格式集合（如 JSON/CSV + zip），但必须预留后续兼容演进路径（与 ADR-029 对齐）。

- **ADR-040｜导出执行语义冻结：导出任务异步化 + reservation_token（额度预留）+ idempotency_key（防重）+ 可取消（Cancel）**
  - 冻结结论：
    - **导出必须异步化**：导出生成属于长耗时/重 IO/可能失败重试的动作，必须以队列/Worker 异步执行（与 ADR-003/004 对齐），不得在同步请求链路中阻塞完成。
    - **额度预留前置（reservation_token）**：导出任务入队前必须由控制面完成额度预留并返回 reservation_token；任务载荷必须携带 reservation_token，Worker 执行前必须强校验，结束必须 Commit/Release（与 ADR-015/016/017 对齐）。
    - **幂等与去重（idempotency_key）**：导出任务必须携带按主账号域生效的 idempotency_key/dedupe_key；入队侧尽可能前置去重，执行侧必须强制幂等，避免 at-least-once 重试导致多份导出包（与 ADR-004/018 对齐）。
    - **可取消（Cancel）语义闭环**：导出必须支持取消入口与状态机闭环；取消/失败/超时必须 Release 预留额度，并对已生成的部分结果进行清理或标记不可下载，避免“用户以为取消但仍在跑/仍在计费/仍可下载”的不一致。

- **ADR-041｜导出权限分层冻结：员工默认可导出“工作所必需范围”；跨范围/批量导出需升级授权（PEP 强制 + 全量审计/计量）**
  - 冻结结论：
    - **分层原则**：导出属于高风险/高成本动作，但不得让主账号管理员成为日常交付瓶颈；因此导出权限采用“默认可用 + 高风险升级授权”的分层模型。
    - **员工默认允许导出工作所必需范围**：员工账号默认可导出其“可编辑/可交付”的资产集合（如当前工作区/当前工作会话/当前交付物等工作必需范围），且必须遵循 ADR-039/040/035 的受控访问、异步、额度预留、幂等、可取消与审计计量闭环。
    - **跨范围/批量/全量导出必须升级权限**：以下类型导出必须由主账号管理员角色执行或由管理员显式授权（capability/授权票据，支持撤销）：
      - 导出整个工作区/大量对象；
      - 导出跨员工/跨主体的资产集合；
      - 导出包含敏感字段/高敏范围的“全量包”；
      - 任何超出“工作所必需范围”的导出。
    - **导出请求必须声明 scope，PEP 强制校验**：导出 API 必须携带明确的导出范围声明（对象类型/集合过滤/字段集/时间窗/是否含敏感字段等）；PEP 必须在主账号隔离上下文下强制校验该 scope 是否被允许，禁止隐式“全导出”。
    - **全量审计/计量**：导出发起/生成/签发/（可观测的）下载必须产生日志化、可审计、可对账的计量事件，并可按主账号与主体归因（与 ADR-013/035/039/040 对齐）。

- **ADR-042｜导出限流与风控冻结：按主账号 + 按主体双维度限流/配额 + 异常峰值分级处置（二次确认/降权/阻断）+ 事件化**
  - 冻结结论：
    - **默认开启双维度限流/配额**：导出相关关键入口（导出发起、导出任务入队、Signed URL 签发、导出下载若可观测）必须同时支持按主账号维度与按主体（员工账号/外部主体）维度的限流/配额控制（与 ADR-012/013 对齐）。
    - **分级处置（progressive enforcement）**：超限/异常峰值必须采用分级响应：轻度返回明确的限流信号并指导重试；中度触发二次确认/重新鉴权/短时冷却；重度可临时降权（例如仅允许“工作所必需范围”的导出）或阻断高风险导出（与 ADR-041 分层模型对齐）。
    - **PEP 强制执行**：限流/配额/风控决策必须在统一 PEP 执行点强制，避免分散在各 API handler 口径不一致；可结合边缘粗粒度防护与服务内精细控制。
    - **事件化审计/计量**：超限、降权、阻断、二次确认触发等风控决策必须产生日志化、可审计、可对账的事件，并可按主账号与主体归因（与 ADR-006/013/035/039/040 对齐）。
  - MVP 收口结论（已对齐）：
    - **MVP 阶段关于“导出”讨论收口到 ADR-042**：导出在 MVP 仅冻结到“对象存储受控访问 + 异步语义 + 权限分层 + 限流风控 + 审计计量”的闭环能力。
    - **更细粒度的导出细节进入 Backlog**：如字段级脱敏/字段集指纹、敏感字段策略矩阵、合规模板化导出等，统一进入后续迭代，不作为当前架构不变量继续扩展。

- **ADR-043｜执行面写入边界冻结：执行面无业务持久化；核心状态与关键对象写入必须回到控制面统一落地**
  - 冻结结论：
    - **执行面（Python / Agent Runtime Hub）默认无业务持久化**：不得直接写入核心业务表/核心对象（如对话资产、交付物、工作流实例、订阅/额度、授权分发、审计/计量事件等）。执行面职责限定为：执行、编排、外部接入（LLM/MCP/A2A/API）、以及流式输出。
    - **所有关键写入收口到控制面（Go / Platform Core）**：执行面产生的结果/状态变更/计量信号，必须通过控制面提供的 API/事件入口回写，由控制面统一执行主账号隔离上下文校验（PEP）、额度闭环（Reservation→Commit/Release）、版本语义（object_version/OCC）、以及审计/计量事件落地。
    - **执行面允许有限持久化（白名单）**：仅允许执行相关的短期中间产物与运行时数据（如任务队列、缓存、运行日志缓冲等），且必须按主账号命名空间隔离并具备回收治理；不得成为业务事实来源。
    - **治理不下沉**：治理逻辑不得散落在 Connector/Adapter 或执行脚本中；任何跨边界/高风险动作必须先获得控制面签发的约束载体（capability/票据/预留凭证）后再执行，并可审计、可追踪、可撤销（与 ADR-009/012/013/016 对齐）。

- **ADR-044｜执行面任务入口冻结：执行面只能执行控制面签发的 Work Ticket/Job Ticket（绑定上下文 + 预留 + 幂等 + 能力集合）**
  - 冻结结论：
    - **唯一入口=Ticket**：执行面（Python）不得接受“裸请求/裸脚本”直接执行；只能执行由控制面签发的 **Work Ticket（在线/流式）** 或 **Job Ticket（离线/异步）**。
    - **Ticket 必须绑定主账号隔离上下文**：Ticket 内必须携带/引用 ADR-008 的主账号隔离上下文（含主体/子账号可选），执行面在关键节点必须强校验上下文一致性；缺失/不一致必须 fail-closed 并事件化。
    - **Ticket 必须绑定 reservation_token（如涉及消耗）**：对任何高成本/会消耗额度的任务，Ticket 必须携带 reservation_token；执行面与 Worker 执行前必须强校验 token 有效性与绑定关系（与 ADR-016/017 对齐）。
    - **Ticket 必须绑定 idempotency_key/dedupe_key**：Ticket 必须携带按主账号域生效的幂等/去重键；执行面必须强制幂等，避免 at-least-once 与重试导致重复副作用（与 ADR-004/018 对齐）。
    - **Ticket 必须绑定 capability set（允许的能力集合）**：Ticket 必须声明允许调用的连接器/外部 Agent/工具集合与动作范围（最小权限），执行面必须据此做调用闸门控制；能力撤销必须可在运行期生效（与 K-003、ADR-009/010/011 对齐）。
    - **Ticket 必须可撤销、可过期、可审计**：Ticket 的签发/使用/拒绝/撤销必须产生日志化、可审计、可对账的事件，并与 trace 关联，支持紧急切断与事后追溯（与 ADR-006/013 对齐）。

- **ADR-045｜Ticket 生命周期治理冻结：TTL + 单次/有限次使用 + 使用回执（receipt）回写；撤销/超时触发取消与额度释放闭环**
  - 冻结结论：
    - **Ticket 必须有 TTL**：TTL 为策略参数（可按任务类型区分）；执行面在关键节点必须校验 TTL，过期必须 fail-closed。
    - **Ticket 必须有使用次数语义**：默认单次使用（one-shot）；如确需多次使用必须显式声明 max_uses 或等价字段，并受控面策略约束。
    - **使用回执（receipt）必须回写控制面**：执行面在至少“开始/完成/失败/取消”关键状态点回写 receipt（含 trace、执行节点、结果摘要/错误码、消耗确认等），用于控制面推进状态机、落审计/计量、并与 reservation_token 的 Commit/Release 对账。
    - **撤销/过期必须触发闭环**：Ticket 被撤销或超时后，执行面必须能在运行期被切断（下一次校验即拒绝继续或尽快停止）；若绑定 reservation_token，必须触发任务取消与 Release 预留额度闭环，避免预留悬挂与不可解释计费。

- **ADR-046｜长任务活性与强制切断冻结：Lease/Heartbeat 续租；拒绝续租即终止；失联自动取消并释放额度**
  - 冻结结论：
    - **长运行任务必须采用 Lease/Heartbeat**：对 Long Job（或任何可能超过短 TTL 的执行）控制面必须维护 lease_expires_at（或等价字段）；执行面/Worker 必须按固定间隔发送 heartbeat 进行续租。
    - **续租是可控开关**：控制面可因 Ticket 撤销/过期、reservation_token 失效、能力集合撤销、主账号风控/超限等原因拒绝续租；执行面一旦续租失败必须尽快停止（fail-closed），并回写 receipt 说明终止原因。
    - **失联自动止损**：超过 lease_expires_at 未续租视为失联；控制面必须将任务标记为超时/已取消，并触发取消任务与 Release 预留额度闭环（与 ADR-017/045 对齐）。
    - **与 Ticket 生命周期对齐**：Lease/Heartbeat 作为 Ticket 运行期治理的补充机制，保证“可撤销/可过期/可切断”在长任务场景下可落地，而不是停留在控制面记录层面。

- **ADR-047｜协作式取消冻结：执行面必须可停（周期性检查取消/续租状态 + 外部调用可超时/可中断 + 终止回执与清理）**
  - 冻结结论：
    - **周期性检查取消信号**：执行面/Worker 的所有长步骤（循环、批处理、分段生成、大文件写入、多次外部调用）必须周期性检查 Ticket 撤销/过期（ADR-045）与 Lease 续租结果（ADR-046）；一旦不可续租/已撤销/已过期必须尽快停止并 fail-closed。
    - **外部调用必须可超时、可中断、可降级**：对 LLM/MCP/A2A/API/对象存储等关键外部调用必须设置超时与重试边界；取消触发后不得继续发起新的外部调用；对在途调用，返回后必须立即终止后续流水线。
    - **终止必须形成 receipt 并清理中间产物**：终止（Cancel/Timeout/Revoked/LeaseDenied 等）必须回写 receipt（与 ADR-045 对齐）；并清理/标记不可用的中间产物（如临时对象存储分片/临时文件/缓存条目），避免“逻辑取消但仍可访问/仍占用资源”。
    - **资源与额度闭环**：任何终止路径必须触发必要的资源释放与额度释放闭环（与 ADR-017/045/046 对齐），避免预留悬挂与不可解释成本。


- **ADR-048｜外部副作用闸门冻结：所有外部副作用调用必须经由 Side-effect Gateway（capability + idempotency + trace），可去重、可审计、可紧急切断**
  - 冻结结论：
    - **副作用边界定义**：凡是会在平台外产生不可逆/难补偿影响的调用均视为外部副作用（例如通知发送、写第三方系统、创建/删除外部资源、支付/扣费、授权分发、触发 RPA 执行等）。
    - **唯一出口=Side-effect Gateway**：执行面内任何 Connector/Adapter/Tool 不得直接调用外部副作用端点；必须经由统一 Side-effect Gateway 作为唯一出口，确保治理口径收口。
    - **强制三件套**：每次副作用调用必须同时携带并校验：
      - capability（允许的动作与范围；不匹配 fail-closed）；
      - idempotency_key（按主账号域生效，网关负责去重/幂等，避免重复副作用；与 ADR-004/018 对齐）；
      - trace（可追溯到 Ticket/主体/链路；与 ADR-006/044 对齐）。
    - **可紧急切断**：控制面撤销 capability、拒绝续租、风控阻断等决策必须在运行期立即对网关生效；网关必须 fail-closed 并回写 receipt/事件以解释拒绝原因（与 ADR-045/046/047 对齐）。
    - **事件化审计/计量**：副作用的请求/响应、去重命中、拒绝、失败重试等必须产生日志化、可审计、可对账的事件，并按主账号与主体归因（与 ADR-013/018 对齐）。

- **ADR-049｜副作用补偿语义冻结：默认 Saga/补偿动作；不可补偿动作必须升级授权/二次确认并提高审计等级**
  - 冻结结论：
    - **副作用必须声明补偿语义**：所有经 Side-effect Gateway 的副作用 action 必须声明 compensation_policy（概念字段）：COMPENSABLE / PARTIALLY_COMPENSABLE / NOT_COMPENSABLE，并随能力描述/策略一并受控与版本化。
    - **可补偿动作必须提供补偿入口**：对 COMPENSABLE action，必须提供补偿动作（undo/compensate 或等价机制），并与正向动作一同走 Side-effect Gateway，强制 capability + idempotency_key + trace；补偿动作必须可重试、可幂等，避免补偿重放造成二次破坏。
    - **不可补偿动作必须升级为更强控制**：对 NOT_COMPENSABLE action，必须使用更强的 capability（更窄 scope、更短 TTL、更严格主体限制），并要求二次确认/强提示（MVP 允许以 UI 二次确认 + reason 记录实现），同时提高审计等级（更完整证据链）。
    - **事件化与对账**：正向动作与补偿动作的请求/响应、去重命中、拒绝、失败重试等必须事件化，并能与 Ticket/主体/主账号/trace 关联，形成可对账证据链（与 ADR-013/048 对齐）。

- **ADR-050｜执行面内不可信执行隔离冻结：Isolation Level 分级；默认受限；高风险能力必须升级隔离与审计（远程外部 Agent 不沙箱但受 egress/数据最小化/能力闸门约束）**
  - 冻结结论：
    - **适用范围澄清**：隔离级别（Isolation Level）适用于“在平台执行面（我们基础设施）内运行”的不可信代码与高风险工具执行（如第三方 Connector/Adapter、托管工具包、解析/转换类工具链、以及任何会触达 Secrets/文件系统/网络的工具执行）。**远程外部 Agent（在对方环境运行）不做沙箱**，但其调用必须受主账号隔离上下文、capability、Side-effect Gateway、以及出站（egress）与数据最小化策略约束。
    - **必须定义 Isolation Level 分级（至少 3 档）**：
      - L0：同进程/同容器（仅限平台自研、低风险、无外部副作用的白名单能力）；
      - L1：容器级隔离 + 默认受限（网络出站最小化/白名单、受限文件系统、最小 secrets 注入、资源配额）；
      - L2+：强化隔离（如 sandbox/microVM 等作为扩展位），用于高敏数据/高风险能力/多方代码托管场景。
    - **默认受限**：凡是“非平台自研或风险未知”的执行，默认至少 L1；允许显式升级到更高隔离等级，禁止默认放宽到 L0。
    - **隔离等级与 capability 风险分层绑定**：涉及敏感数据访问、广域网络访问、不可补偿副作用、跨系统写入、或更高权限 capability 的任务，必须升级隔离等级并提高审计等级（与 ADR-048/049 对齐）。
    - **运行期可强制与可切断**：PDP 决策隔离等级；PEP/执行面强制落地；当 capability 撤销/拒绝续租/风控阻断发生时，隔离相关策略（如 egress allowlist、secrets 注入）必须在运行期立即生效并 fail-closed。

- **ADR-051｜远程外部 Agent 出站与数据最小化冻结：egress 默认 deny+allowlist；禁止内网/元数据；请求与响应按 capability 裁剪/脱敏；全链路可审计**
  - 冻结结论：
    - **egress 默认 deny + 显式 allowlist**：凡由平台执行面代发起的外部请求（HTTP/API/MCP 工具访问等）必须默认拒绝，并按主账号/按 capability/按连接器显式配置 allowlist 后才允许出站。
    - **显式禁止高风险目标**：禁止访问云元数据服务与 link-local/localhost/内网网段等高风险目标，防止 SSRF 借道访问内部资源或窃取云环境凭证；目标校验必须在统一执行点强制，禁止分散在各 connector 内各自实现。
    - **数据最小化为默认**：对远程外部 Agent 的请求与响应必须按 capability 做字段级裁剪/脱敏/摘要（最小必要集）；敏感字段默认不出站、不落盘，确需出站必须显式升级能力与审计等级。
    - **与副作用闸门联动**：远程外部 Agent 产生的任何外部副作用仍必须经 Side-effect Gateway（ADR-048/049），并受 capability、idempotency_key、trace 与可撤销/可切断约束；不得因“远程执行”而绕过治理。
    - **全链路事件化审计**：allow/deny、命中策略、被阻断原因、裁剪/脱敏命中等必须事件化，可关联主账号/主体/Ticket/trace，用于追溯与风控复盘。

- **ADR-052｜MVP 最小数据分级冻结：三档（PUBLIC/INTERNAL/SENSITIVE）+ 默认策略挂钩（日志/出站/落盘/导出），作为 SSOT 可版本化**
  - 冻结结论：
    - **MVP 仅冻结最小三档分级**：PUBLIC / INTERNAL / SENSITIVE（或等价命名），避免一开始引入过细的四级/矩阵治理，但必须有统一口径支撑 ADR-051 的“敏感字段默认不出站/不落盘”。
    - **分级必须绑定 4 个硬控制点的默认策略**：
      - **日志（Logging）**：SENSITIVE 禁止明文入日志；必须脱敏/哈希/截断；INTERNAL 默认不记录 payload 明文；PUBLIC 可按需。
      - **出站（Egress）**：SENSITIVE 默认不出站；INTERNAL 出站需显式 allow（与 ADR-051 对齐）；PUBLIC 可按 allowlist。
      - **落盘（Persistence）**：SENSITIVE 默认不落盘或仅以受控形态落盘（例如仅存引用/摘要/指纹）；INTERNAL 落盘需明确用途与保留策略；PUBLIC 常规落盘。
      - **导出（Export）**：SENSITIVE 导出必须升级授权/二次确认并提高审计等级（与 ADR-041/042/048/049 对齐）；INTERNAL 导出需满足“工作所必需范围”或显式授权；PUBLIC 常规导出。
    - **分级与策略必须 SSOT 化并可版本化**：数据分级定义与默认策略（含脱敏规则、字段裁剪规则、导出字段集）必须作为可机读配置/契约纳入仓库与版本控制；禁止散落在各 Connector/工具链里各自实现。
    - **默认最严 + 显式放行**：对未标注/无法判定的数据默认按 INTERNAL 或 SENSITIVE 处理（策略参数可配置），放宽必须显式声明并产生审计事件。

- **ADR-053｜数据分级标注机制冻结：对象级默认分级 +（可选）字段级覆盖；未标注默认最严；标注变更版本化并事件化审计**
  - 冻结结论：
    - **对象级默认分级（必须）**：对核心对象/资源类型必须定义 default_classification（PUBLIC/INTERNAL/SENSITIVE）；系统在日志/出站/落盘/导出等策略决策时必须可查询该默认分级。
    - **字段级覆盖（扩展位）**：机制必须存在，用于对少量高风险字段启用更严格分级与规则（如 secrets/PII/财务字段等）；MVP 不要求全字段标注，但允许按需逐步补齐。
    - **未标注默认最严**：对未知/未标注 payload，默认按 INTERNAL 或 SENSITIVE（策略参数）处理，避免漏标导致外泄；放宽必须显式。
    - **标注与规则必须版本化（SSOT）**：对象默认分级、字段覆盖、脱敏/裁剪/导出字段集规则必须作为可机读配置纳入仓库与版本控制；禁止分散在各 Connector/工具链。
    - **变更必须事件化审计**：任何标注/规则变更必须产生审计事件（谁改的、改了什么、何时生效、影响范围），支持回滚与追责；关键路径在变更生效时应产生可观测信号以便验证。

- **ADR-054｜敏感数据进入 LLM 上下文冻结：SENSITIVE 默认不进入 Prompt/RAG/Memory；确需进入必须显式 capability + 脱敏/裁剪 + 审计计量；默认不入长期记忆**
  - 冻结结论：
    - **默认禁止**：被标注为 SENSITIVE 的数据不得直接进入任何 LLM 上下文（system/user prompt、RAG chunk、memory、以及任何工具输出拼接入 prompt 的内容），以降低 prompt injection 与意外外泄风险。
    - **例外必须显式（三件套）**：确需将敏感数据用于推理（例如财富规划引用具体数值）时，必须同时满足：
      - **显式 capability**：允许“将某类敏感字段用于推理”的最小范围、最短 TTL、可撤销；
      - **脱敏/裁剪策略**：字段级掩码/区间化/摘要化，仅提供最小必要集；
      - **事件化审计/计量**：记录主账号上下文、触发的 capability、命中哪些脱敏/裁剪规则、出站到哪个模型/远程外部 Agent（与 ADR-051/052/053 对齐）。
    - **默认不入长期 Memory**：进入上下文的敏感片段默认只作为临时态参与本次推理，不进入长期记忆/可检索记忆；若确需记忆必须升级授权并明确 retention（与 ADR-052/053 对齐）。
    - **与出站/副作用闸门联动**：任何对模型/远程外部 Agent 的出站仍受 ADR-051（egress+最小化）约束；任何外部副作用仍受 ADR-048/049 约束，不得因“推理需要”绕过治理。

- **ADR-055｜统一 Guardrail 冻结：LLM 输出与工具调用必须经 Guardrail（结构/契约校验 + 敏感泄漏检测 + 动作级策略校验），不合规降级/拒绝并事件化**
  - 冻结结论：
    - **统一执行点（PEP 强制）**：任意 LLM 输出进入系统关键路径（进入工具、进入导出、进入外部、进入持久化/记忆等）必须先过 Guardrail；任意工具调用（含参数）必须先过 Guardrail。
    - **MVP 最小三类校验（足够覆盖主风险）**：
      - **结构/契约校验**：输出/工具参数必须满足 schema（JSON Schema/等价契约）；不满足则拒绝或降级为纯文本，禁止“半结构化字符串”直接驱动系统动作。
      - **敏感泄漏检测**：命中 SENSITIVE（与 ADR-052/053/054 对齐）默认拦截/脱敏/降级，禁止敏感信息继续流转或出站。
      - **动作级策略校验**：任何外部副作用仍必须走 Side-effect Gateway（ADR-048/049）并满足 capability/idempotency/trace；缺失任一要素必须 fail-closed。
    - **不合规处理必须可解释且事件化**：返回机器可读 reason_code（含拦截/降级/拒绝原因），并产生日志/审计/计量事件，可关联主账号/主体/Ticket/trace。
    - **扩展位（非 MVP 强制）**：对高风险或不可补偿动作可要求人类审批/二次确认，但不改变 Guardrail 作为统一前置的主路径。

- **ADR-056｜Guardrail SSOT 与版本化冻结：规则/Schema/敏感模式/脱敏策略必须版本化（可灰度/可回滚），每次决策可回溯到规则版本**
  - 冻结结论：
    - **Guardrail 规则必须 SSOT 化并版本化**：至少包含输出/工具参数 schema、敏感模式（规则/分类器/模式集）、脱敏/裁剪策略、动作级策略（哪些动作必须走 Side-effect Gateway 等）；必须有明确的 policy_version。
    - **每次放行/拦截/降级都必须可回溯**：Guardrail 决策事件必须记录 policy_version、rule_id/reason_code、命中类型（结构不合规/敏感泄漏/动作不允许…），并关联主账号/主体/Ticket/trace（与 ADR-006/008/013 对齐）。
    - **规则变更必须可灰度与可回滚**：MVP 不要求复杂策略平台，但必须支持至少一种维度的灰度（按环境/按比例/按主账号白名单等）与快速回滚到上一版本，避免误拦/漏拦造成全局事故。
    - **变更必须事件化审计**：规则从 vX→vY 的发布/启用/回滚必须产生审计事件（发布者/时间/范围/生效版本），支持追责与复盘。

- **ADR-057｜Guardrail 执行位置收口冻结：必须在控制面 PEP 统一落地（网关/核心服务）；执行面只调用，不得各自实现一套策略口径**
  - 冻结结论：
    - **统一 PEP 执行点（控制面）**：Guardrail 的强制执行必须收口到控制面 PEP（API Gateway / Platform Core 的关键入口/出口），覆盖至少：对外 HTTP/WS、对执行面签发与回写（Ticket/receipt）、对外出站与副作用闸门等关键 choke point。
    - **执行面禁止形成第二套策略口径**：执行面（Python / Worker / Connector）不得内置独立策略集或独立“放行/拦截”决策；其职责是调用控制面 Guardrail/PEP 的统一能力并遵循结果。
    - **允许防御性断言但不改变权威口径**：执行面可以做“防崩溃/防脏数据”的轻量 schema 断言与输入净化（defensive checks），但不得作为权威策略决策来源；权威策略版本与 rule_id/reason_code 仍以控制面 PEP 产出的决策事件为准。
    - **与版本化/灰度/审计对齐**：ADR-056 的 policy_version、灰度/回滚、事件化审计必须以控制面 PEP 为主入口落地，避免多点部署导致规则漂移。

- **ADR-058｜统一出站出口冻结：对外出站必须经 Egress Gateway（allowlist + 目标校验 + 速率/配额 + 事件化），执行面不允许直连外网**
  - 冻结结论：
    - **统一出口（不可绕过）**：平台对外出站（LLM Provider、Remote Agent、MCP 外部工具、第三方 API）必须统一经过 Egress Gateway 作为出口 PEP；执行面（Python/Worker/Connector）不允许直连公网。
    - **默认 deny + 显式 allowlist**：出站目标必须匹配 allowlist（可按主账号/按 capability/按连接器维度配置），未匹配必须 fail-closed（与 ADR-051 对齐）。
    - **目标校验与硬禁止（SSRF 防线）**：必须硬禁止 localhost、link-local、内网网段、云元数据服务等高风险目标；目标校验必须在统一出口强制，禁止分散在各 connector 内各自实现。
    - **按主账号限流/配额与成本治理**：出站必须纳入 ADR-012 的限流/配额（可含 burst），并产生日志化计量事件用于归因与对账（与 ADR-013 对齐）。
    - **事件化审计与可追溯**：对每次 allow/deny、命中策略、阻断原因、目的域名/IP、上游 Ticket/trace 等必须事件化（与 ADR-006/008/056 对齐），并支持灰度/回滚与紧急切断。
    - **与副作用闸门协同**：Egress Gateway 负责“出站目标与流量治理”；任何外部副作用仍必须经 Side-effect Gateway（ADR-048/049），不得因统一出口而弱化副作用治理边界。

## 4. 结论对后续文档/实现的约束（Implementation Guardrails）

- **G-001｜任何请求/事件/任务必须携带隔离上下文**：HTTP / WebSocket / Async Job / DB / Redis / VFS 等边界都必须显式携带并恢复主账号/子账号隔离上下文；禁止隐式默认。
- **G-002｜开发期门禁必须自动化**：CI 中必须存在可机读的规则与测试，用于阻断“未注入隔离上下文的 DB/缓存/队列访问”与“高风险动作未走 PEP/capability”的变更。
- **G-003｜运行期兜底必须可审计**：关键隔离与授权决策必须产生可关联的审计与可观测性信号（至少可追踪到请求/事件/任务级别）。
- **G-004｜全链路 trace context 必须传播与可关联**：HTTP / WebSocket / Async Job / 对外 MCP/A2A/API 调用必须提取/注入 trace context；关键日志必须可关联到 trace。
