## 1. 引用规范（所有下游产物必须遵守）

适用范围：所有下游产物（PRD/架构/协议/测试/运维/研究文档），以及代码命名与测试用例命名。

### 1.1 术语引用格式（强制）

- 下游产物在引用任何产品域术语时，必须使用：`[TERM-XXX] 中文术语`
- 禁止只写中文术语而不带 TERM-ID。
- 若需在同一句中多次出现该术语，可在首次出现处写全格式，后续可仅写 `[TERM-XXX]`（但不允许只写中文术语）。

**示例：**
- 正确：员工进入 [TERM-WS-013] 工作会话 后开始任务。
- 错误：员工进入“工作会话”后开始任务。（缺少 TERM-ID）

### 1.2 禁止事项（强制）

下游产物（PRD/技术文档/测试文档/运维文档/研究文档/代码命名/测试用例）中禁止出现：

1) **重定义术语**：不得在下游文档中给出新的“术语定义/同义词表/边界说明”，只能引用本 SSOT。  
2) **复制粘贴定义**：不得把 SSOT 里的定义整段复制到下游文档；下游只允许写引用（TERM-ID + 中文术语），必要时补充“该上下文的使用说明”，但不得改变边界。  
3) **发明新产品域名词**：新增名词必须先在本 SSOT 中新增 TERM（并补齐定义/边界/变更记录），再由下游引用。  
4) **出现冲突时自行裁决**：当下游与 SSOT 或其他 SSOT 文档冲突时，不得在下游文档“就地修正口径”；必须回到 SSOT/上游文档按冲突解决流程处理。

### 1.3 代码与测试命名的最小约束（强制）

- 若代码/测试用例名称使用产品域概念，必须能映射到一个 TERM-ID（建议在注释或用例描述中标注 `[TERM-XXX]`）。
- 不得在代码/测试中引入未经 SSOT 登记的新产品域名词。

## 2. 术语清单（唯一口径）

> 本节分组仅用于阅读与检索，不与 PRD 文件结构绑定；未来无论 PRD 拆分或合并，本表结构不需要随之调整。
>
> 字段说明：
> - **定义**：一句话讲清楚。
> - **边界**：必须写“含/不含”。
> - **同义词/禁用词**：用于避免多叫法漂移；PRD 中出现禁用词应改为本表“中文术语”。

| TERM-IA-032 | 策略执行点 | Policy Enforcement Point (PEP) | **产品定义**：负责“在关键入口拦住并执行裁决”的统一拦截点。<br>**工程定义**：不做策略推理；负责拦截请求、组装上下文、调用 PDP、执行 Permit/Deny/Obligations，并写入审计与回执。 | **含**：拦截、执行裁决、落审计；**不含**：绕开 PDP 的隐式放行、执行面接受裸请求直接执行。 | 同义：网关拦截点；**禁用**：自己决定、软拦截（仅告警不阻断）。 | PEP 可落在 API Gateway / Agent Runtime / Connector Gateway 等边界点。 |

| TERM-G-010 | 副作用网关 | Side-effect Gateway | **产品定义**：所有会改变外部状态的动作都必须走“更严格的统一出口”，默认更谨慎、更可追溯。<br>**工程定义**：所有写回/下单/发送/删除等不可逆操作的强治理入口；默认 fail-closed；可要求二次确认/条件允许（obligations）；必须生成 receipt 与 reason_code。 | **含**：写回/变更类操作；**不含**：纯读取查询（可归入 Egress）。 | 同义：Side-effect Gated；**禁用**：直接写回、静默执行、无回执。 | 拒绝必须返回可机读 reason_code，并写入审计。 |
| TERM-G-011 | 回执 | Receipt | **产品定义**：每次对外操作的“可追溯凭证”：做了什么、结果如何、谁批准/为何被拒。<br>**工程定义**：append-only 证据记录；含请求摘要、决策摘要（允许/拒绝/原因）、结果状态、trace 关联、外部对象引用；可用于对账、复盘、重放校验。 | **含**：可检索、可关联、可审计；**不含**：仅日志打印、不可关联的零散记录。 | 同义：副作用回执；**禁用**：仅日志、口头确认。 | Receipt 与审计/计量事件应可在 Insights 中回溯展示。 |
| TERM-G-012 | 追踪上下文 | Trace Context | **产品定义**：把一次业务动作从前端到后端、到异步任务、到外部调用全部串起来的“链路编号”。<br>**工程定义**：W3C Trace Context：traceparent（必选）+ tracestate（可选）；要求在 HTTP/WS/Async/Receipt/Outbound 全链路传播。 | **含**：跨服务传播与关联；**不含**：只在单服务内打点、异步链路丢失。 | 同义：W3C Trace Context；**禁用**：无兼容策略的自定义 trace header。 | “Trace Context Everywhere” 是系统可观测性不变量。 |
| TERM-G-013 | 拒绝原因码 | reason_code | **产品定义**：系统拒绝/失败的“统一原因标签”，便于统计、告警与复盘。<br>**工程定义**：稳定枚举、机器可读；用于 PDP/PEP/Gateway/Runtime 错误语义统一；不可用自然语言 message 代替。 | **含**：稳定枚举；**不含**：随意拼字符串、仅 message、仅 HTTP status。 | 同义：error_code；**禁用**：临时字符串、仅 message。 | 示例：quota_exceeded / policy_denied / consent_required / replay_detected。 |
| TERM-G-014 | 事实事件 | Facts | **产品定义**：系统中所有关键行为与结果的“事实记录”，用来驱动报表与复盘，避免口径漂移。<br>**工程定义**：不可变事实事件流；聚合/派生数据必须可由 Facts 重建；禁止以聚合表/缓存作为唯一真相。 | **含**：不可变、可回放；**不含**：以报表为准、无来源派生口径。 | 同义：facts-first；**禁用**：以聚合为准、以缓存为准。 | 授权分发/回收、计量、审计、receipt 等应进入 Facts 供 Insights 复盘。 |
| TERM-G-015 | 证据链 | Provenance / Lineage | **产品定义**：回答“这个结论从哪来、怎么得出、谁在何时为何允许/拒绝”的最小依据集合。<br>**工程定义**：最小字段集合（trace_id、source_refs、query_plan_summary、acting_subject、policy_decision、receipt[]、inputs_hash…）；用于 grounding、审计复盘与可重建。 | **含**：可回溯与可重建所需最小信息；**不含**：黑盒结论、只存最终结果。 | 同义：lineage；**禁用**：无依据/黑盒结论。 | 派生属性必须携带最小 provenance（否则标记为不可验证）。 |
| TERM-G-016 | 语义映射层 | Semantic Mapping Layer | **产品定义**：把企业数据/文档/事件统一映射为“对象与关系”的可理解结构，方便检索、引用与复用。<br>**工程定义**：对象/关系语义建模层；提供 Object View / Link View、schema_version、派生属性承载与可查询接口；不是仅文档向量召回。 | **含**：对象/关系建模与版本；**不含**：仅向量检索的 chunk 召回、把 UI 展示当语义层。 | 同义：semantic layer；**禁用**：临时口径层、随意映射。 | 用于 grounding 与可组合查询；对标“对象类型/属性/关系”的建模方式。 |
| TERM-G-017 | 对象视图 | Object View | **产品定义**：把一个业务对象（如客户/合同）以统一结构呈现，便于被引用与被行动。<br>**工程定义**：typed JSON + schema_version；包含稳定属性与可版本化派生属性（带 provenance）；可 stale 标记。 | **含**：类型化对象态、版本；**不含**：仅文本摘要、无 schema 约束的临时 JSON。 | 同义：typed object view；**禁用**：随便一段 JSON、无版本对象态。 | AI 产生的结论写入 derived_attributes，并必须附 provenance。 |
| TERM-G-018 | 关系视图 | Link View | **产品定义**：把对象之间的关系（如“客户-合同-订单”）以统一方式表达，便于遍历与组合分析。<br>**工程定义**：object_link(src_object_id, link_type, dst_object_id, attrs... )；可查询/可遍历/可聚合；不是仅文本相似度共现。 | **含**：类型化关系、可遍历；**不含**：无类型弱关系、仅相似度当关系。 | 同义：relationship view；**禁用**：随意关联、无类型关系。 | link_type 必须来自 [TERM-G-019] 语义对象注册表（SOR）中已注册的 [TERM-G-022] 关系类型（Link Type）。 |
| TERM-G-019 | 语义对象注册表 | Semantic Object Registry (SOR) | **产品定义**：一份受控、可版本化的“语义类型目录”，用于统一对象/关系/动作的命名、字段口径与治理锚点，为后续 Ontology 演进预留路径。<br>**工程定义**：Control Plane 的最小语义元数据注册表；至少包含 Object Type / Action Type / Link Type；读写必须受 policy_check 保护，默认 fail-closed；变更需可审计并可发布更新事件。 | **含**：类型 key、版本、schema/属性口径、状态（active/deprecated/disabled）、治理引用（如 SideEffectProfile）；**不含**：具体对象实例数据、图推理/复杂关系查询实现。 | 同义：type registry；**禁用**：随意 JSON schema、无版本“约定俗成”。 | SOR 变更后驱动 SDK/codegen 与缓存失效；为“类 Palantir Ontology”提供稳定锚点。 |
| TERM-G-020 | 对象类型 | Object Type | 受控的对象“类型定义”，描述一类对象允许有哪些属性、属性类型与版本兼容规则。 | **含**：type_key、schema_version、属性定义、兼容性与弃用策略；**不含**：单条对象实例（实例用 ObjectRef/Object View 表达）。 | 同义：entity type；**禁用**：把 UI 字段当类型、无版本字段。 | 示例：customer、contract、workflow_run 等类型的统一口径。 |
| TERM-G-021 | 动作类型 | Action Type | 受控的动作“类型定义”，描述一类动作的输入/输出 schema、幂等约束与副作用治理口径。 | **含**：action_key、input/output schema、幂等要求、SideEffectProfile 引用；**不含**：一次具体执行记录（执行用 ActionRef/Receipt 表达）。 | 同义：operation type；**禁用**：把实现函数名当动作类型、无治理口径的“随便写回”。 | 示例：send_email、create_ticket、place_order 等。 |
| TERM-G-022 | 关系类型 | Link Type | 受控的关系“类型定义”，描述两类对象之间允许建立的关系及其约束。 | **含**：link_key、src/dst Object Type 约束、方向性/基数、边属性 schema（可选）；**不含**：图推理与复杂遍历算法实现。 | 同义：relationship type；**禁用**：无类型弱关联、仅相似度当关系。 | 示例：customer->contract、contract->invoice。 |
| TERM-G-023 | 对象引用 | ObjectRef | 指向某个“类型化对象实例/对象视图”的稳定指针，用于跨系统/跨事件引用对象。 | **含**：object_type_key + object_id（可选 revision/etag）；**不含**：对象 payload（payload 用 Object View 单独查询/获取）。 | 同义：object pointer；**禁用**：把对象 JSON 直接嵌进事件/消息。 | 在交互事件/审计/回执中引用对象，避免传大对象正文。 |
| TERM-G-024 | 动作引用 | ActionRef | 指向某次“动作意图/动作执行”的稳定指针，用于幂等、追踪与回执关联。 | **含**：action_type_key + action_id（可选 receipt_id）；**不含**：动作执行细节全文（细节以 Receipt/Audit 事实为准）。 | 同义：action pointer；**禁用**：只靠日志字符串定位一次动作。 | ActionRef 可在提交后先返回，Receipt 后到或异步补齐。 |
| TERM-G-025 | 副作用档案 | SideEffectProfile | 描述某类 Action Type 的“副作用治理画像”，用于 Side-effect Gateway 的默认策略与义务（obligations）判定。 | **含**：风险级别、是否需要人工确认/二次确认、允许的出站通道、必须记录字段、回执要求；**不含**：具体审批 UI/工作流实现细节。 | 同义：side-effect policy profile；**禁用**：静默执行、无回执写回。 | 示例：高风险动作要求 human_review + idempotency_key + 强制 receipt。 |

## 3. 变更与新增流程
- 新增术语：
  1) 新术语按模块前缀 `IA/WS` 或 `G/MP` 进行编号；`IN` 为预留前缀，只有在本 Glossary 补齐对应模块术语并发布版本后才允许启用与引用。
  2) 补齐定义/边界/禁用词；
  3) 在 PRD 中仅做引用。
- 修改术语：
  - 必须补充“变更原因/影响文档清单”，并更新版本号与更新时间。
- 任何新增/修改术语，必须同时更新：本表对应行（定义/边界/禁用词）+ 变更记录（日期/版本/摘要/影响范围）；并在 PRD/L2/L3 文档中仅以 TERM-ID 引用，不得复制定义。
